4. Programmatic SEO Assessment

The 940-page library follows the same pattern as Zapier's app-pair pages or G2's comparison pages. Zapier targets "connect App A to App B." Proxuma targets "what does [metric] look like in Power BI for [data source]?" Lower volume per page, but far less competition.

What Google rewards in this space

  • Specificity over breadth -- a page about "SLA compliance by priority in Autotask" outranks a generic "MSP KPIs" listicle
  • Technical depth -- DAX formulas, actual chart descriptions, metric definitions signal expertise
  • FAQPage schema -- Google surfaces FAQ rich snippets aggressively for "how to" MSP queries
  • Internal linking density -- the 6-link related reports system is well above industry average
  • Data attribution -- citing specific dataset sizes (135K alerts, 67K tickets) signals real analysis vs. template content

Risk: thin content

The pSEO model breaks if pages share too much boilerplate. Each page needs 300+ unique words beyond the shared template (morph bar, related reports, CTA). Current pages average ~2,100 words with substantial unique analysis per report -- this is well above the threshold. The risk is in any future batch-generated pages that skip the narrative sections.

5. Technical SEO Audit

Title tag pattern (verified live)

Format: [Report Title]: [Keyword Phrase] | Proxuma Power BI

Example: "Autotask SLA Performance by Priority Level: First Response & Resolution Compliance for MSPs - Proxuma"

Strong keyword-first positioning, 50-70 characters, brand suffix. No changes needed.

Meta description pattern (verified live)

Format: "Power BI report: [Title]. Live MSP analytics built on [source] data."

Some pages use richer descriptions with specific data points. The generic template version is weaker -- pages with custom descriptions that cite specific numbers (135K alerts, 264 companies) get higher CTR.

Hreflang implementation (critical gap)

Live verification confirms: NL pages have no hreflang tags. The HTML lang attribute is correctly set to nl-NL, and JSON-LD includes inLanguage: nl-NL, but the actual <link rel="alternate" hreflang="..."> tags are missing. Without these, Google has to guess the EN/NL relationship. For 420+ NL pages, this is a major indexing risk.

Internal link architecture

Category affinity model with 7 groups: Ticketing, Costs, Resources, Projects, Client Reports, Upsell, Security. Each report gets 6 related reports via add-related-reports.py. Cross-category links follow predefined affinity rules (e.g., Ticketing links to Client Reports, Resources, Costs). This is well-executed -- rare for automated systems.

Bart recommended expanding from 4 to 6 links per page. The system already does 6. The CSS grid supports 3x2 layout.

Sitemap analysis (verified live)

WordPress sitemap at /sitemap_index.xml contains 4 child sitemaps: post, page, faq, knowledge-base. The page-sitemap includes ~180 EN report URLs and ~120 NL report URLs. The remaining ~640 pages from the repo are either not yet published to WordPress or published but not in the sitemap.
