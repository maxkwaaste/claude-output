# DAX Query Audit — Power BI Report Pages

**Date:** 2026-04-12  
**Scope:** 828 report HTML files (434 EN + 394 NL) in `~/ClaudeCode/powerbi-reports/`

---

## Summary

The DAX queries embedded in every report page are **fabricated**. They were generated by AI without querying the actual Power BI schema. While many table names happen to match real tables, the column names and query logic are invented. None of the DAX queries execute successfully in Power BI.

---

## Bug Fixed This Session

**Double click handler on DAX toggles** — every `.dax-toggle` div had both an inline `onclick` and a JS event listener on the child `.dax-trigger`. Click events bubbled up, both handlers fired, and they cancelled each other out. The toggles appeared broken (nothing opened on click). Fix: removed the inline `onclick` from all 828 files. The JS listener at the bottom of each file now works correctly.

### Table Name Typos

Some reports use `BI_Nable_*` (lowercase 'a') but the schema uses `BI_NAble_*` (capital 'A'). DAX is case-insensitive for table names so this wouldn't cause failures, but it indicates the queries weren't sourced from the actual schema.

## Column Name Analysis

Even where table names are correct, the column names used in DAX queries are fabricated. Examples:

### Report: 28-Day Backup Heatmap
- Uses: `BI_Datto_Backup_Jobs[is_successful]`, `[company_name]`, `[snapshot_month]`
- Reality: Table doesn't exist at all

### Real table column mismatches (BI_Datto_Rmm_Alerts)

## Options

1. **Strip all DAX toggles** — Remove the entire `.dax-toggle` div from every report. Clean and honest. Reduces file size. Visitors never see broken queries.

2. **Replace with disclaimer** — Keep the toggle UI but swap the DAX code for a note: "This report was AI-generated. Executable DAX queries for your dataset can be generated on request." Preserves the visual structure.

3. **Leave as-is behind toggle** — The toggles now work (bug fixed), but the queries are fiction. Low risk since most visitors won't copy-paste into Power BI Desktop, but undermines credibility if anyone does.

4. **Regenerate with real DAX** — For each report, query the actual schema and write valid DAX. This is the most work (~2,500 queries) but would make the reports genuinely useful. Could be done incrementally per report topic.
