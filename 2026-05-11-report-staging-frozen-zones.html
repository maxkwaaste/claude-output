1. Current Situation

The Proxuma/powerbi-reports repo holds 872 remediated HTML files (449 EN, 423 NL) on main, ready for WordPress deployment. Eight remediation stages are complete. Stage 9 (live deploy) and Stage 10 (post-push verification) are blocked on visual verification.

What exists

  • dev.proxuma.io resolves to GitHub Pages (maxkwaaste.github.io). Returns HTTP 200. Serves raw HTML without the WordPress theme shell (no site header, no nav, no footer, no theme CSS).
  • Report files are self-contained: each has its own inline CSS, chrome wrapper (morph bar, search hero, pipeline, overview, CTA), teal-frame document, JSON-LD, and JavaScript. They render their own visual structure standalone.
  • WordPress wraps these in the Proxuma theme via page.php, adding the site header/nav and footer around the report content.

What's missing

Infrastructure gap Three files referenced by the /powerbireport command spec don't exist in the repo or locally:
  • templates/teal-frame-css.html (the locked CSS template)
  • GOLDEN-STANDARD.html (the visual reference file)
  • deploy_reports_lib.py (the deployment library with wrap_with_chrome())

The CSS is baked into each report file individually. There is no single-source CSS file to update across all 872 reports. This makes visual fixes a batch operation every time.

2. Staging Environment: Three Options

The goal is to see reports rendered inside the actual Proxuma WordPress theme, exactly as visitors will see them. GitHub Pages can't do this. Three paths forward:

Option A
SG Staging
SiteGround WordPress staging clone. Full theme fidelity. 10 min setup via Site Tools UI.
Option B
Theme Shell
Inject WP theme header/footer HTML into GitHub Pages via build script. 80% fidelity.
Option C
Local WP
Local WordPress with Proxuma theme. Full fidelity but setup overhead.

Option A: SiteGround Staging (recommended)

SiteGround Site Tools has a one-click staging feature that clones the production WordPress site. This gives you the real theme, all plugins (ACF, Yoast, SG CachePress, If-So), and the actual CSS/JS stack. Reports deployed here render identically to production.

Setup steps:

  1. Log into SiteGround Site Tools for proxuma.io
  2. WordPress > Staging > Create Staging Copy
  3. SiteGround gives you a staging URL (typically stg-proxuma-io.sgXX.siteground.eu or similar). You can optionally CNAME dev.proxuma.io to point there instead of GitHub Pages.
  4. SSH into the staging environment and deploy reports via $wpdb->update()
Key limitation SiteGround staging is a full clone, so deploying 872 pages means 872 $wpdb->update() calls on the staging database. This is the same operation you'll run on production, so it doubles as a deployment rehearsal. However, SiteGround's staging environment shares server resources with production. Run the batch during off-peak hours.

Option B: Theme Shell on GitHub Pages

Scrape the Proxuma theme's header and footer HTML from a live page, then wrap each report file in that shell via a build script. Push to GitHub Pages. Gets you ~80% visual fidelity: the nav, footer, and theme fonts render, but dynamic WordPress features (If-So geolocation, SG CachePress, ACF) won't work. Good enough for visual QA of report-internal elements, not for testing theme interactions.

Option C: Local WordPress

Set up LocalWP or wp-env with the Proxuma theme cloned from production. Full fidelity on your machine, but requires installing the theme, plugins, and importing content. More setup than SiteGround staging, and harder to share with others for review.

Recommendation

Use SiteGround staging (Option A) as the primary visual verification environment. It's the only option that gives 100% theme fidelity with zero theme setup work. Re-point dev.proxuma.io to the staging URL only if you want a clean domain; the SiteGround staging URL works fine as-is.

3. Deployment Pipeline for Staging

The deploy_reports_lib.py referenced in the /powerbireport spec doesn't exist. But the reports in the repo already have chrome wrappers baked in (they're full-page HTML, not teal-frame-only). So deployment is simpler than the spec implies: push the HTML content directly into WordPress page post_content via $wpdb->update().

Batch deploy sequence

  1. Create page stubs. Each report needs a WordPress page under the correct parent (3299 for EN, 3495 for NL). Pages that already exist on production will exist on staging after the clone. New reports from the remediation (if any don't have WP pages yet) need to be created first.
  2. Map files to page IDs. Use hreflang-mapping.json and the existing WP page list to build a slug-to-page-ID mapping.
  3. Push HTML via SSH. For each file, extract the content inside <body>...</body> and push it via: 
    $wpdb->update('wp_posts', ['post_content' => $html], ['ID' => $page_id]);
  4. Flush cache. wp cache flush && wp sg purge after the batch.
  5. Spot-check 10 pages across different report categories before doing a full visual pass.
Protected pages The deploy script must skip page IDs 3299, 3495, 3401, 5655, 3389, 4887. These are gallery pages and editorial content. The existing PROTECTED_PAGE_IDS safeguard needs to be in whatever deploy script you build.

4. Visual Error Taxonomy

Once reports are on staging, you'll encounter visual errors in roughly six categories. Ordered by likelihood and severity:

Class What Happens How to Detect Prevalence
CSS class drift Subagents invented CSS classes that don't exist (.bar-chart, .kpi-card, .stat-card, .metric-card). Elements render as unstyled plain text. Grep all files for class names not in the V5 stylesheet. The /report-audit command has the class mapping. High. Known from the Feb-Mar audit. Remediation stages may not have caught all instances.
Div imbalance Extra or missing closing </div> tags. Breaks container nesting, causing content to spill outside the teal frame or collapse sections. HTML parser validation. Stage 8 QA fixed 135 div balance issues but LLM generation is probabilistic. Medium. 135 were fixed; residual cases likely.
WP theme CSS conflict WordPress theme CSS overrides inline report styles. Common culprits: h1/h2 tag styles, a link colors, table resets, img max-width. Visual comparison: same report on GitHub Pages (no theme) vs. staging (with theme). Differences are theme conflicts. Medium. Reports use .prx-report scoping but generic selectors in the theme can still win specificity battles.
Color/font deviation Wrong hex codes (e.g., #1B365D vs #1a2744), wrong font families (Inter/Montserrat instead of Open Sans), wrong font weights. Regex scan for non-standard color values and font-family declarations across all files. Low-medium. Template was consistent but individual session regenerations may have drifted.
Responsive breakage KPI rows don't stack on mobile, morph bar overlaps content, tables overflow horizontally, teal frame padding collapses. Resize browser to 375px width on staging. Check KPI row, tables, and morph bar. Low. Most reports share the same responsive CSS, but custom chart CSS may lack media queries.
Content artifacts Leftover template placeholders, generic filler phrases that survived Stage 1 cleanup, duplicate FAQ JSON-LD blocks, "TODO" markers. Grep for known artifact patterns: "We pulled the numbers", "Schedule Recurring Review", "TODO", duplicate FAQPage schemas. Low. Stages 1-8 targeted these specifically, but some edge cases may remain.

5. Frozen Zones: The Visual Contract

A Power BI report page has distinct visual zones. Some must look identical across all 872 reports, regardless of content. Others vary by report topic but follow fixed styling rules. This section defines which is which, so when you're reviewing visual errors, you know exactly what "correct" looks like for each zone.

REPORT PAGE ANATOMY [ WP Theme Header + Nav ] ← WordPress (not in report HTML) [ Morph Bar (fixed, z-10000) ] ← FROZEN: #115e58, 50px, word cascade [ Search Hero ] ← FROZEN structure, VARIABLE question text Breadcrumb / Badge / Question / Source pills Shelf card (title, meta, CTA) [ Pipeline Section ] ← STYLE-FROZEN, content varies [ Report Overview ] ← STYLE-FROZEN, content varies [ Teal Frame ] ← FROZEN: gradient, 50px pad, 12px radius [ Doc Letterhead ] ← FROZEN: logo, rule, meta layout [ Doc Title Block ] ← FROZEN style, VARIABLE text [ Doc Body ] [ KPI Row ] ← STYLE-FROZEN: 4-col grid [ Sections 1.0, 2.0... ] ← STYLE-FROZEN: numbering, headers [ Tables ] ← STYLE-FROZEN: borders, padding [ Patching/Compliance Bars ] ← STYLE-FROZEN: colors, heights [ DAX Toggles ] ← STYLE-FROZEN: dark bg, JetBrains Mono [ Narrative text, findings ] ← Content only, styling from .doc-body [ Doc Footer ] ← FROZEN: layout, text [ CTA Block ] ← FROZEN (outside teal frame) [ Related Reports ] ← STYLE-FROZEN, cards vary [ Series Navigation ] ← STYLE-FROZEN if present [ WP Theme Footer ] ← WordPress (not in report HTML)

Legend: FROZEN = pixel-identical across all reports. STYLE-FROZEN = same CSS rules, different content. CONTENT-ONLY = only the text varies; styling inherited from parent container.

6. Frozen Zone Specifications

Each zone below lists exactly what must remain constant. When reviewing a report for visual correctness, check these values. If any deviate, the report has drifted.

Zone 1: Morph Bar

Pixel-frozen. Identical on every report.
  • position: fixed; top: 0; height: 50px; z-index: 10000
  • Background: solid #115e58 (no gradient, no border)
  • Trigger: activates when .report-frame-outer top reaches viewport top
  • Animation: slide-down (scaleY) + word cascade (translateY(-18px) scale(0.8) blur(2px) to rest)
  • Word delay: 35ms stagger per <span class="w">
  • Pills: integration names + "All reports" back link (teal #5eead4)
  • Font: Open Sans italic, 0.88rem, white
FROZEN

Zone 2: Search Hero

Structure frozen, question text varies.
  • Gradient: linear-gradient(160deg, #0f766e 0%, #134e4a 40%, #0f2240 100%)
  • Dot grid overlay, green glow at bottom
  • "AI-Generated Report" badge with pulsing dot
  • "THE QUESTION WE ASKED" label
  • Question text: Playfair Display italic (this varies per report)
  • Source pills: badge colors per source (PSA blue, RMM yellow, etc.)
  • Shelf card: white background, overlapping hero bottom
FROZEN STRUCTUREVARIABLE TEXT

Zone 3: Teal Frame

Pixel-frozen. The signature visual element.
  • Background: linear-gradient(180deg, #0f766e 0%, #115e58 50%, #134e4a 100%)
  • Padding: 50px on all sides (same visual weight as morph bar height)
  • Border-radius: 12px
  • Inner .doc: white background, border-radius: 4px, box-shadow
  • Never use #00D9A5 green for the frame. Always the teal gradient.
FROZEN

Zone 4: Document Letterhead

Pixel-frozen. Top of the white document.
  • Left: Proxuma SVG logo + "AI-Generated Power BI Report" type label
  • Right: Date, Source badges, Scope line
  • Bottom border: 3px solid #1B365D (navy rule)
  • Padding: 36px 48px 28px
  • Meta text: 0.74rem, #64748b, strong labels in #1B365D
FROZEN

Zone 5: KPI Row

Style-frozen. Layout identical, values vary.
  • Grid: grid-template-columns: repeat(4, 1fr), no gap, shared borders
  • Background: #fafbfc
  • Border: 1px solid #e2e8f0, border-radius: 6px
  • Label: 0.66rem, uppercase, #94a3b8
  • Value: Open Sans 800, 1.6rem, #1B365D
  • Note colors: green #10B981, red #ef4444, amber #f59e0b
STYLE-FROZEN

Zone 6: Section Structure

Style-frozen. Numbered sections with consistent headers.
  • Section numbers: Open Sans 800, 0.82rem, #0f766e (teal)
  • Section title: Open Sans 700, 1rem, #1B365D
  • Section padding: 28px 0, border-bottom 1px solid #f1f5f9
  • Doc body padding: 0 48px
  • Structure: .doc-section-head with .doc-section-num + .doc-section-title
STYLE-FROZEN

Zone 7: DAX Toggles

Style-frozen. Dark code blocks with copy buttons.
  • Toggle class: .dax-toggle (NOT .inline-dax which is a drift marker)
  • Code font: JetBrains Mono
  • Dark background code block
  • Copy button: "Copy Query" (EN) / "Kopieer Query" (NL)
  • Exactly 3 toggles per report, placed at the bottom of data-heavy sections
  • Function: prxCopyDAX(btn)
STYLE-FROZEN

Zone 8: Tables

Style-frozen. Clean standard styling, data varies.
  • Header: no dark background (unlike the output-page .matrix style). Clean borders.
  • Cell padding consistent across all tables
  • No invented wrapper classes (.data-table, .stats-table are drift markers)
  • Correct class: bare <table> inside .doc-body, styled by the scoped CSS
STYLE-FROZEN

Zone 9: Patching/Compliance Bars

Style-frozen. Horizontal stacked bars with fixed colors.
  • Classes: .patch-row, .patch-bar-wrapper, .patch-segment
  • NOT: .bar-chart, .bar-row, .bar-fill (those are drift markers from subagents)
  • Colors: .bar-danger (red), .bar-warning (amber), .bar-success (green)
  • NOT: .critical, .warning, .success (generic names that drift)
STYLE-FROZEN

Zone 10: Document Footer + CTA

Pixel-frozen. Same text, same layout on every report.
  • Doc footer: "Proxuma Power BI - AI-Generated Report" + "Generated in under fifteen minutes via MCP"
  • CTA block: sits OUTSIDE the teal frame
  • Both are chrome elements, not content
FROZEN

7. Drift Detection: Class Whitelist

The single biggest source of visual errors is AI-generated CSS class names that don't match the actual stylesheet. Here's the authoritative mapping between drift markers (wrong classes) and correct classes:

Drift Marker (wrong) Correct Class Notes
.bar-chart, .bar-row, .bar-fill .patch-row, .patch-bar-wrapper, .patch-segment Horizontal stacked bars
.kpi-grid, .kpi-card, .kpi-value .doc-kpi-row, .doc-kpi, .doc-kpi-value KPI metric row
.section-card Unwrap into .doc-section No wrapper card needed
.inline-dax .dax-toggle Different HTML structure too
.critical, .danger .bar-danger Red status color
.warning, .amber .bar-warning Amber status color
.success, .green .bar-success Green status color
.data-table, .stats-table Plain <table> in .doc-body Styled by scoped CSS, no class needed
.metric-card, .stat-card .doc-kpi Inside .doc-kpi-row

8. Validation Strategy

A three-layer approach, from cheapest to most expensive:

Layer 1: Automated (run on all 872 files)

  • Div balance check. Parse HTML and verify opening/closing div counts match. The qa-validate.py script in the repo already does this.
  • Class whitelist scan. Grep for known drift markers (the table above). Flag any file containing them.
  • Color audit. Extract all hex colors from inline styles. Flag any that aren't in the V5 palette (#0f766e, #115e58, #134e4a, #1B365D, #2c3e50, #fafbfc, etc.).
  • Font audit. Grep for font-family declarations. Flag Inter, Montserrat, or any font not in the spec (Open Sans, DM Serif Display, JetBrains Mono).
  • Artifact scan. Grep for known filler phrases, "TODO", duplicate FAQPage schemas.

Layer 2: Screenshot comparison (sample of 20-30 reports)

  • Pick 3-4 reports from each category (SLA, CSAT, resource, finance, device, project, client, queue, pipeline).
  • Capture full-page screenshots on staging via Microlink API or Playwright.
  • Compare frozen zones visually: morph bar, teal frame, letterhead, KPI row, footer.
  • This catches theme CSS conflicts that automated scans miss.

Layer 3: Manual spot-check (5-10 reports)

  • Open reports in a real browser on staging.
  • Scroll through: does the morph bar trigger correctly? Do DAX toggles open/close? Do tables render without horizontal overflow?
  • Test at 375px width (mobile) for at least 3 reports.
  • This catches JavaScript issues and responsive breakage.

9. Working with Claude on Visual Fixes

When you find visual errors on staging, here's how to communicate them efficiently so Claude can fix them without introducing new drift:

What to tell Claude

  1. Which zone is broken. Use the zone numbers from this document: "Zone 5 (KPI Row) is wrong on this report."
  2. What it looks like vs. what it should look like. A screenshot is best. Or: "The KPI values are showing in gray instead of navy #1B365D."
  3. Whether it's one report or many. "This affects sla-performance and 3 other SLA reports" vs. "All 872 files have this."

What Claude should NOT do

  • Rewrite the entire CSS block "to be cleaner." The CSS is intentionally copied verbatim from the template. Optimization = drift.
  • Add new CSS classes. Use only the existing class vocabulary.
  • Change frozen zone values (colors, paddings, font sizes) to "look better." The spec is the spec.
  • Touch doc-body content when fixing chrome/frame issues. Scope repairs to the broken zone only.

The Golden Rule

When fixing visual errors, the CSS template is the source of truth, not what "looks right" to the model. If a report's CSS diverges from the template, the fix is always to restore the template values, never to update the template to match the report.

10. Infrastructure To-Do Before Staging Deploy

  1. Create templates/teal-frame-css.html. Extract the CSS from one known-good report (e.g., en/proxuma-io-sla-performance-post.html) and lock it as the canonical template. Every future report generation and every batch fix should reference this file.
  2. Create GOLDEN-STANDARD.html. Pick one visually verified report as the reference. This is what "correct" looks like.
  3. Build deploy_reports_lib.py. Needs: push_via_tempfile(), PROTECTED_PAGE_IDS safeguard, slug-to-page-ID mapper, batch deploy with progress logging. The remediation scripts (qa-validate.py, fix-chrome-markers.py) are good starting points for the file I/O patterns.
  4. Build visual-audit.py. Combines Layer 1 checks: div balance, class whitelist, color audit, font audit, artifact scan. Run before every deploy.
  5. Set up SiteGround staging. Manual step via Site Tools UI.
