Content Platform for MSP Consultants

How we went from a PDF with MSP problems to a full product spec in one brainstorming session. This document walks through the design decisions, mockups, and final architecture for the Proxuma Content Platform.

The Starting Point

We have a team of 5 Dxfferent consultants with deep MSP and Autotask expertise. Their knowledge is the source of all our content, but there's no structured way to capture it. AI can transform content into blog posts, LinkedIn posts, and articles, but it can't be the source. We need a way to farm our experts for their knowledge in a way that's convenient for them.

Key Principle AI cannot be the source of content. It can only transform what our experts write. The platform's job is to make contributing as frictionless as possible.

Design Decisions

The brainstorming session covered several key decisions. Here's how each one landed.

Decision 1: Two entry paths

Max
"If they want to do free form they should be able to. Claude can then categorize. If they want to pick a topic this should also be available as an option."

Result: Consultants can either browse the 63 topics and claim one, or hit "Vrije tekst schrijven" to dump raw text without picking a topic. Free-form submissions get categorized by Claude later.

Decision 2: Dual filter system

We explored three options for how consultants navigate the topic list. The question: do consultants think in domains ("I know about PSA") or content types ("I want to write a case study")?

Three Options Mocked Up in Excalidraw Option A: Filter by content category only (Bewijs, Educatie, Markt...)
Option B: Filter by MSP Journey domain only (Data Driven, Technology...)
Option C: Both filter rows, independently selectable

Result: Option C. Two filter rows: DOMEIN (grey pills) and TYPE (colored pills matching the contentbibliotheek palette). Consultants can combine filters or use either independently.

Decision 3: Full-width editor

When a consultant clicks a topic, we open a full-width editor view (not an inline accordion). The left side is the writing area with auto-save. The right side shows context: the hook template, category badge, and relevant MSP Journey problems/frustrations/questions for that domain.

Decision 4: Storage and hosting

Max
"Use an URL like content.dxfferent.com instead. You have SiteGround and WP access to dxfferent.com."

Result: Standalone app at content.dxfferent.com on a separate SiteGround subdomain. Completely isolated from WordPress. SQLite database via PHP PDO. No auth (5 internal users), noindex for search engines.

Decision 5: Phased approach

The full vision includes AI transformation, content calendar, LinkedIn integration, and analytics. We broke it into three phases to ship value fast.

Architecture

File Structure

content.dxfferent.com/
  index.html          -- SPA (topic board + editor views)
  api.php             -- REST API (CRUD on SQLite)
  seed.json           -- 63 topics + MSP Journey context
  data/
    content.db        -- SQLite (auto-created on first request)
    .htaccess         -- Deny from all
  .htaccess           -- URL rewriting, noindex, cache headers

Database Schema

topics (id, title, hook, category, domain, subdomain, status, claimed_by, created_at)
submissions (id, topic_id, author, raw_content, category, domain, created_at, updated_at)
authors (id, name, created_at)

API Endpoints

MethodEndpointPurpose
GET/api/topicsList all topics (filters: category, domain, status)
GET/api/topics/:idSingle topic with submissions
PATCH/api/topics/:idUpdate status, claimed_by, title, hook
POST/api/submissionsCreate submission (body: topic_id, author, raw_content)
PATCH/api/submissions/:idEdit content inline
GET/api/statsCounts by status, category, domain

Three Phases

Now Phase 1: Content Harvesting
Topic board with dual filters, full-width editor, free-form input, SQLite storage, visual status tracking. Ship to content.dxfferent.com by April 30.
Next Phase 2: Transformation + Calendar
Claude transforms raw content into blog posts, LinkedIn posts, and articles. Content calendar for scheduling. Publishing status board with color-coded visuals.
Later Phase 3: Publishing + Analytics
LinkedIn API integration for direct posting. dxfferent.com publishing. Page view tracking, LinkedIn metrics, and content performance analytics to steer topic selection.

Portability to Proxuma

The architecture is designed for later integration into the Proxuma platform (Laravel + Nuxt 3) via the claude-harness manifest system.

  • seed.json feeds a Laravel database seeder directly
  • API contract maps 1:1 to Laravel resource routes
  • SQLite schema translates to a standard Laravel migration
  • Frontend is decoupled via /api/*: swap the backend, keep the UI

Bottom Line

One brainstorming session produced a complete spec for a content platform that solves a real problem: getting expert knowledge out of consultants' heads and into a structured library. Phase 1 ships tomorrow. The rest builds on top without rework.
