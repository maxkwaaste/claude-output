The session started with a request to write an implementation plan for the existing Phase 2.5 design spec. The spec had already been produced in a prior brainstorming session and contained detailed requirements for kanban, metrics, derivatives, and export. This session's job was to turn that spec into step-by-step build instructions an agent could follow without additional context.

What We Started With

Design spec (pre-existing)

430-line design document at docs/superpowers/specs/2026-04-29-phase-2.5-design.md covering four features with exact SQL, HTML, CSS, and PHP code. Written during a brainstorming session earlier the same day.

Codebase state

Architecture constraint Everything lives in two files. No build step, no dependencies, no splitting. All CSS and JS inline in index.html. All API routes in one api.php switch statement. Deployed via SCP to SiteGround.

The Four Features

1. Kanban Board
Five-column board (Idee, Schrijven, Review, Gepland, Gepost) with HTML5 drag-and-drop. Topics placed by combining topics.status with calendar.status via subquery. Toggle between list and kanban views, persisted to localStorage. Mobile: columns stack, drag replaced with dropdown.
UI API New status: review
2. Fake Performance Data
Deterministic metrics (impressions, clicks, engagement rate, comments) generated from entry ID as seed. Ranges vary by content_type + platform. Seeded on posted entries, NULL on manually-posted ones. "Genereer demo data" button fills NULLs. Dashboard gets performance cards. Calendar entries show inline metrics.
Schema API UI
3. Content Derivatives
New derivatives table with foreign key to submissions. Each submission gets derivatives based on topic category (bewijs gets 4 formats, tools gets 2). API serves flat list, nested tree, and aggregate stats. Board cards show derivative count. Dashboard gets format breakdown with progress bars.
Schema API UI
4. CSV + Google Sheets Export
Export panel on calendar tab with date range picker. CSV endpoint uses early-exit pattern (outputs before JSON Content-Type header). Sheets sync via Apps Script webhook stored in new settings table. UI shows the Apps Script snippet for setup.
API UI External: Google Apps Script

Cross-cutting Changes

Soft Delete (archived_at)
No user content is ever hard-deleted. archived_at DATETIME added to submissions, calendar, and derivatives. Every DELETE endpoint becomes an UPDATE. Every SELECT adds WHERE archived_at IS NULL. Roughly 15 queries affected across api.php.
Schema Migrations
Uses PRAGMA table_info to detect missing columns, then ALTER TABLE ADD COLUMN. Handles both fresh seeds (new DB) and existing databases. No migration framework needed for SQLite.

The plan was written using the superpowers:writing-plans skill. Each task contains exact file paths, line numbers, complete code blocks, test commands with expected output, and commit instructions. No placeholders, no "implement similar to above".
