Verdict: RAM The machine cannot absorb 2 GB for Docker without shedding existing load. The 30 Claude sessions and Ollama's 14B model are the two biggest consumers. One or both need to shrink.

CPU Budget

The blueprint asks for 1 dedicated core. Current load average of 7.80/8 cores shows the machine is already near saturation in peak moments. The cron jobs create burst patterns (07:00 and 11:00 are the busiest windows). Adding four more Claude-driven cron jobs at 09:07, 10:15, 14:30, and 08:00 Monday would stack onto existing peaks.

Three Paths Forward

Option A: Shed Load to Make Room
Lowest cost, highest friction
  • Downgrade Ollama from 14B to 7B model (saves ~600 MB resident)
  • Kill idle Claude sessions (30 is excessive; likely 20+ are orphaned tmux sessions from prior cron runs). Target: max 5 concurrent.
  • Skip Docker, run OpenClaw gateway as native Node.js process under PM2 (saves ~500 MB Docker overhead)
  • Result: ~1.5-2 GB freed. Enough to run the gateway.
Option B: Time-Slice the Machine
No hardware change, smart scheduling
  • Unload Ollama models when OpenClaw is active (they reload in ~5 sec)
  • Schedule OpenClaw cron jobs in gaps: 09:07, 10:15, 14:30 are already clear windows
  • Set OLLAMA_MAX_LOADED_MODELS=0 during SDR windows, reload after
  • Cap Claude sessions: add cleanup cron that kills sessions older than 2 hours
  • Result: gateway runs in borrowed headroom, but Ollama unavailable during those windows.
Option C: Upgrade Hardware
Most headroom, costs money
  • M1 Pro with 16 GB was right for 2024 workloads. The current 30-session + Ollama + always-on services load has outgrown it.
  • 32 GB M-series Mac Mini would handle everything with room to spare (~$900 refurb)
  • Alternatively: move Ollama to M4 Air (16 GB, lower background load) and keep M1 Pro as the automation server without local LLM.

Recommendation

Go with Option A + elements of B. The machine can handle OpenClaw if you:

  • Clean up orphaned Claude sessions (likely saves 500+ MB immediately)
  • Run OpenClaw as native PM2 process, not Docker (skip the 2 GB container overhead)
  • Downgrade to qwen2.5-coder:7b or unload when not actively coding on M1 Pro
  • Add a session reaper cron: kill Claude processes idle > 2 hours

This gets you the always-on gateway + all four new cron jobs within existing hardware. The paused CRM-migration tasks (email triage, morning briefing, etc.) will need the session cleanup done before they can safely come back online too.
