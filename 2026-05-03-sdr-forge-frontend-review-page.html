What was built

The SDR Outreach Review page lets a reviewer work through a daily batch of AI-generated email drafts. Each draft can be approved (sends via MS Graph), skipped (with optional feedback for AI tuning), or snoozed. The reviewer can edit subject and body before sending, coach the AI with free-text feedback, and see the AI's reasoning for each draft.

The page lives at /forge/sdr/review inside the existing Forge section of the NeXum frontend. It uses the standard app layout (sidebar provided by the platform) and follows all project conventions: Tailwind utility classes, semantic design tokens, VCard/VButton/VIcon primitives, i18n via vue-i18n, and WCAG AA compliance.

Architecture decisions

State management

All state lives in the page component (review.vue) using ref and computed. No Pinia store was added. The batch is fetched once via useAsyncData, then draft statuses are mutated locally on approve/skip/snooze (optimistic update). The API call fires in parallel. If the API fails, the finally block resets the busy flag but the local status stays (acceptable for a review workflow where retries are manual).

No sidebar

The Claude Design prototype included a full sidebar. That was stripped. The NeXum app layout already provides a sidebar via AppSidebar. The page only renders the main content area.

No custom fonts

The prototype bundled Inter woff2 files. Those were not copied. Inter is already loaded via Google Fonts in nuxt.config.ts.

Category colors via inline styles

The 5 category colors (green, blue, yellow, purple, red) are defined in SDR_CATEGORIES and applied via :style bindings. This is intentional: category colors are domain data, not theme tokens, and they need to work in both light and dark themes without Tailwind class proliferation.

Keyboard shortcuts

Handled in the page's onMounted with a global keydown listener. Skips when focus is in an input/textarea/select. Shortcuts: A=send, S=skip, Z=snooze, J/K=navigate, Shift+A=approve all.

What was NOT built (by design)

  • Toast component -- uses Nuxt UI's built-in useToast() composable instead of a custom toast stack
  • Navigation registration -- the page is accessed via URL; nav entries are configured in config/nexum-modules.php on the backend
  • "Regenerate batch" action -- button is rendered but not wired (no backend endpoint for regeneration yet)
  • Undo via API -- undo is local-only (resets the draft status in the reactive array); backend doesn't have a revert endpoint
