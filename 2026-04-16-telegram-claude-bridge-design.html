Status

Approved for implementation planning. All five design sections validated with Max in the 2026-04-16 brainstorming session. Source spec committed to ~/ClaudeCode/claude-native/docs/superpowers/specs/2026-04-16-telegram-claude-bridge-design.md.

Background

The Claude Native setup on M1 Pro already runs an always-on Claude session via the com.proxuma.claude-telegram LaunchAgent. A previous session fixed permission prompts by adding --dangerously-skip-permissions, --permission-mode bypassPermissions, and --model claude-opus-4-6 to the launch command. See ~/ClaudeCode/claude-native/TELEGRAM-BOT-FIX-FINDINGS.md.

Three things still need building:

  1. A /clear mechanism that works from the Telegram chat
  2. File delivery from M1 Pro back to Max's phone
  3. A confirmation protocol so Max sees the session was actually cleared

The Telegram plugin only registers /start, /help, /status. Rather than patch the plugin cache (overwritten on plugin updates, three versions shipped in 8 days), the design keeps all new code outside the plugin.

Architecture

Telegram chat  ↔  Bot API  ↔  telegram plugin  ↔  Claude (always-on, channels)
                                                        │
                                                        ├─ helper scripts (new)
                                                        │   ├─ telegram-clear.sh
                                                        │   └─ telegram-send.sh
                                                        │
                                                        └─ SessionStart hook ── curl "ready" ──▶ Telegram

LaunchAgents (unchanged):
  com.proxuma.claude-telegram  (runs start-telegram.sh)
  com.proxuma.claude-watchdog  (respawns on crash, 60s check)

Two shell helpers, two rules added to the M1 Pro project-scoped CLAUDE.md, one SessionStart hook entry in M1 Pro ~/.claude/settings.json. Nothing inside the plugin cache, so plugin updates do not break the bridge.

Components

1. ~/.claude/helpers/telegram-send.sh (new, ~25 lines)
Generic bot API helper
Usage: telegram-send.sh "text" or telegram-send.sh --doc /path/to/file. Reads token from ~/.claude/channels/telegram/.env, chat ID hard-coded 8764211796. Calls sendMessage or sendDocument via curl. Exits non-zero on failure.

Machine guard at top: checks ~/.claude/projects/-Users-maxdekwaasteniet/CLAUDE.md for "This Machine: M1 Pro". If absent, exits 0 silently. Required because ~/.claude/ is scp-synced across M1 Pro and M4 Air.
2. ~/.claude/helpers/telegram-clear.sh (new, ~10 lines)
Clear orchestrator
Calls telegram-send.sh "Session clearing, fresh Claude booting..." then launchctl kickstart -k gui/$(id -u)/com.proxuma.claude-telegram. launchd respawns the agent within 5-30s.
3. M1 Pro project-scoped CLAUDE.md (edit)
Bot behavior rules
Adds two rules under a new "Telegram Bridge" heading: (a) on /clear, run the helper and do not reply further; (b) file delivery: substantial outputs via output-page skill, everything else via telegram-send.sh --doc.
4. SessionStart hook in ~/.claude/settings.json (M1 Pro)
Post-boot "ready" signal
Runs telegram-send.sh 'Fresh session ready.' once per Claude boot. Machine guard in the helper prevents it firing on M4 Air.

Data flow

Flow A — Normal message (unchanged)

  1. Max types a request
  2. Plugin forwards to Claude via channels transport
  3. Claude replies via ctx.reply

Flow B — /clear

  1. Max types /clear
  2. Plugin forwards as plain text (no bot.command('clear') handler)
  3. Claude matches the CLAUDE.md rule, runs Bash: ~/.claude/helpers/telegram-clear.sh
  4. Script sends "Session clearing, fresh Claude booting..."
  5. Script kickstarts the LaunchAgent, old Claude dies
  6. launchd respawns within 5-30s
  7. New Claude boots, SessionStart hook sends "Fresh session ready."
  8. Max's next message hits the fresh session

Flow C — File / artifact delivery

  1. Claude creates e.g. /tmp/report.md
  2. If substantial (report, plan, spec, > ~15 lines formatted) → output-page skill → GitHub URL → reply with URL
  3. Otherwise → telegram-send.sh --doc /tmp/report.md → Telegram attachment
  4. Max taps on his phone

Error handling

Failure modeMitigation
Respawn gap (5-30s, up to 60s)Bot API buffers updates. Messages typed during the gap reach the new session. Pre-kill message tells Max to wait.
SessionStart hook silentCheck ~/ClaudeCode/claude-native/logs/telegram.log. Fallback: move the "ready" curl into a trap-based wrapper in start-telegram.sh.
Claude ignores /clear ruleLow risk given --dangerously-skip-permissions and the existing Draft Approval Protocol precedent. Manual fallback: SSH to M1 Pro and kickstart directly.
telegram-send.sh failsNon-zero exit. Claude surfaces the Bash failure in chat. No silent swallowing.
Plugin auto-updateNo impact. Nothing touched inside ~/.claude/plugins/cache/.
/clear mid-taskIn-flight tool call finishes first, then the clear runs. Acceptable.

Testing

Validation sequence Four manual checks after deployment, in order. If all four pass, the bridge works.
  1. Baseline ping. Send hi. Expect a reply, no permission prompts.
  2. File delivery. Ask for a 1-line text file at /tmp/test.txt sent as a document. Expect a Telegram attachment, tappable, with content "hello".
  3. Big-output delivery. Ask for a 400-word project summary via the output-page skill. Expect a github.com/maxkwaaste/claude-output/... URL.
  4. /clear round-trip. Tell Claude "my favorite color is teal." Wait for ack. Send /clear. Expect "Session clearing..." within ~1s and "Fresh session ready." within ~30s. Ask "what is my favorite color?" → Claude should say it does not know.

Out of scope

  • Patching the Telegram plugin
  • Changing the cron-based task runner (run-task.sh, still paused)
  • Re-authenticating MS365 (separate, on-demand)
  • Telegram voice calls (Bot API does not support bot voice)
  • iOS Shortcut fallback (rejected in favor of /clear in chat)
  • Multi-user / multi-chat support

Decisions log

DecisionRationale
CLAUDE.md rule over plugin patchSurvives plugin auto-updates (three versions in 8 days).
Two confirmation messages (pre-kill + post-boot)Unambiguous evidence the respawn completed. Silence is ambiguous.
Script-level machine guard instead of settings.local.json fork~/.claude/ is scp-synced identically. Without the guard, SessionStart fires on M4 Air and spams Telegram whenever Max runs cc.
No iOS ShortcutMax wants to type /clear in chat, not tap a shortcut icon.
