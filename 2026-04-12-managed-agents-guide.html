Claude Managed Agents

What you can build, how it works, and a ready-to-paste setup prompt

What It Actually Is

Managed Agents is Anthropic's hosted agent runtime. You define an agent config (model, tools, system prompt), point it at an environment (sandbox settings), and Anthropic runs the full agent loop for you. Each run is a "session" with its own container where bash, file ops, and code execution happen server-side. You just stream events and send messages.

The mental model: Agent (config, create once) + Environment (sandbox, create once) = Session (a run, create per task). You never manage infra. Anthropic handles the loop, the container, prompt caching, context compaction, and extended thinking.

Key Capabilities
Custom Tools (Client-Side)
Define tools the agent can call, but YOUR app handles the execution. Agent fires agent.custom_tool_use, you respond with user.custom_tool_result. Good for keeping API keys on your side, calling internal services, or running actions that need human approval.
MCP Server Integration
Connect to any MCP-compatible service (GitHub, Linear, Notion, Asana, Slack, etc.). Declare the server on the agent, store OAuth credentials in a vault, attach the vault to the session. Anthropic handles token refresh.
GitHub Repository Mounting
Clone repos into the session container at startup. Agent can read, edit, commit, and push. For PR creation, also connect the GitHub MCP server. Multiple repos per session supported.
File Upload and Download
Upload files via the Files API, mount them as session resources. Agent writes outputs to /mnt/session/outputs/ which you can list and download after the session. Max 999 file resources per session.
Skills
Attach domain-specific expertise: pre-built (xlsx, docx, pptx, pdf) or custom skills you upload via the Skills API. Max 64 per agent. The agent loads them on-demand when relevant.
Permission Policies
Control tool execution: always_allow (auto-run) or always_ask (pause and wait for your approval before running). Applied per-tool for fine-grained control.
Agent Versioning
Every update to an agent config creates a new immutable version. Pin sessions to a specific version for reproducibility. Roll back if a prompt change regresses. A/B test versions side-by-side.

Things to Know
  • First-party only. Not available on Bedrock, Vertex, or Foundry. Direct Anthropic API only.
  • Agent first, always. You must create an agent before creating a session. No inline agent config on the session.
  • Create once, run many. Never call agents.create() per-request. Store the ID, reuse it.
  • Stream first, send second. Open the SSE stream before sending messages, or you might miss early events.
  • MCP auth goes through vaults. Credentials are stored separately, never inline on the agent config.
  • Custom tool results are required. If the agent calls a custom tool, it blocks until you respond.
  • Session outputs. Agent writes to /mnt/session/outputs/. Brief indexing lag (~1-3s) after idle before files appear in list.
