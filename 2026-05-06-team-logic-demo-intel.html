Pain Points

  • No portfolio view. Autotask has zero portfolio-level project planning capability. "Nonexistent" per Philip.
  • Flat templates. Engineers build projects where every task has the same start/end date with no dependencies. Moving dates becomes a full-day job for someone.
  • Schedule management nightmare. Constantly moving schedules across all projects, even those that haven't started. Someone spends entire days doing nothing but updating dates.
  • 12-week project lead time. Way further out than desired. Know they need to add capacity, but also recognize it's partly a utilization and efficiency problem.
  • Shared resources. No clean separation between service delivery and project work. Engineers get pulled in both directions.
  • No dedicated PM. Account managers and the owner are splitting project lead duties alongside their actual jobs. "Doesn't really work out that well, unfortunately."
  • Client-facing project visibility. Want to share project status with clients (especially co-managed) and assign tasks to clients. Currently no good solution outside Autotask licensing.

3. Demo Topics Covered

  • Resource/dispatch calendar - drag-and-drop service calls, color-coded by type (purple = ticket, blue = project task, green = service call)
  • Outlook integration - bidirectional sync, service calls sent to engineer calendars with hyperlinks to project tasks, reads Outlook appointments back
  • Project Gantt view - phases, tasks, service calls hierarchy. Inline editing of fields, time entries, notes, attachments
  • Capacity screen - heat map showing planned work per resource (shades of blue = utilization, red = overbooked), includes PTO and internal activities
  • Bulk edit feature - shift-select multiple tasks, move by days/weeks, service calls move with them
  • Template system - master template with modular phases (admin, prep, kickoff, POC, delivery, adoption). Select which phases apply. Consulting-provided master template included.
  • Shift planner module - rotation scheduling for service desk coverage with capacity impact
  • Proposal mode - create projects in proposal stage to see calendar impact before committing
  • Power BI integration - dashboards for time logging compliance, effective hourly rate per customer, revenue per contact, project hours (estimated vs. planned vs. worked)
  • AI-powered reports - customer-facing project reports generated in 5 minutes with client branding, plus internal reports with margin data and verifiable DAX queries
  • MCP server architecture - Philip spotted and asked about it. Understands the concept of plugging AI into Power BI data.
  • Dxfferent consulting (Guide as a Service) - sprint-based consulting every 2 weeks, process maps for every MSP process, Autotask configuration recommendations, master template

6. Objections & Concerns

Process before tooling
Philip clearly stated that process needs to be fixed first regardless of tool choice. "No matter what tool we end up looking at, our overall process is where we're gonna have to target first." This is not an objection to Proxuma specifically, but it could delay a purchase decision.
Existing commitments
Already invested in Sea Level OPS (through Pax 8) and Giant Rocketship. Need to evaluate whether to continue with Sea Level or move to something "more modern." Sunk cost psychology could slow the switch.
Task ordering limitation
The Autotask API not exposing task order means flat-date projects will look messy in Proxuma. Philip's projects currently have this problem. He needs to clean up templates before Proxuma can shine.
Manual ticket dispatch
Proxuma's ticket dispatch is manual vs. Giant Rocketship's auto-dispatch. Philip wants both tools to coexist. Max confirmed this works fine.
Hiring timeline dependency
Philip wants to align the tool rollout with hiring a dedicated PM. "It would be nice to have that person starting around the same time." If the hire gets delayed, the deal could stall.
