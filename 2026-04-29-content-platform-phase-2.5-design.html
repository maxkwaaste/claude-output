Starting Point

The Proxuma Content Platform was already live at content.dxfferent.com with 3 tabs (Board, Dashboard, Kalender), a PHP REST API backed by SQLite, and a vanilla JS SPA. 64 MSP topics seeded, 5 consultants with demo submissions, 15 calendar entries, and an auto-planner algorithm. A code review had already been applied (CORS lockdown, input validation, database indexes).

The brief: design and build four features -- a kanban board, fake performance metrics, content derivatives view, and CSV/Google Sheets export. All within the existing vanilla JS + PHP + SQLite stack, no frameworks.

Design Decisions

Eight clarifying questions were asked one at a time. Each resolved a key architectural choice.

Question 1
Kanban placement: 4th tab or replace/toggle within Board?
Three options presented: replace Board (A), toggle within Board (B), separate 4th tab (C). Decision: B -- toggle within Board. Keeps tab count tight, acknowledges kanban and list are the same data, consultants switch views as needed.
Question 2
Kanban stages vs current topic statuses?
Current DB has 3 statuses (open, claimed, written). Kanban needs 5 columns (Idee, Schrijven, Review, Gepland, Gepost). Two approaches: expand status enum (A) or add separate kanban_stage column (B). Decision: A -- expand status, derive Gepland/Gepost from calendar JOINs. Single source of truth, no state duplication.
Question 3
Content derivatives: attach to submissions or topics?
Derivatives represent output formats (LinkedIn post, blog, email snippet) generated from raw content. Decision: A -- attach to submissions. The submission is the actual human-written content that gets transformed. Preserves the audit trail of who wrote what. User emphasized: "quality is paramount, we can't have AI building on AI with content."
Question 4
Fake performance data: stored in DB or computed on the fly?
Decision: A -- stored in DB. Four new columns on calendar table (impressions, clicks, engagement_rate, comments). Simple SQL aggregations for dashboard, CSV export reads columns directly, schema is ready for real analytics data. User requirement: "fake data needs to be deletable instantly and the architecture must be built for real data."
Question 5
CSV export scope: current week, date range, or full dump?
User pivoted: "ideally it talks to Google Sheets, if possible a live connection." Decision: Google Apps Script webhook for live sync + CSV download as fallback. Apps Script is ~20 lines, no OAuth needed, webhook URL stored as a platform setting. Both options available in an inline export panel on the calendar tab.
Architectural Principle "No content should ever be destroyed, just archived. It's only text after all." All DELETE endpoints become soft deletes via archived_at column. Demo data wipe clears metrics and archives derivatives but never destroys submissions, calendar entries, or topics.

Feature Designs

1. Kanban Board
Toggle within Board tab
Five columns: Idee (open) → Schrijven (claimed) → Review (new status) → Gepland (has planned calendar entry) → Gepost (has posted calendar entry). HTML5 drag-and-drop, no library. Moving to Gepland opens calendar modal with date picker. Filters apply across all columns. Mobile: columns stack, drag replaced with dropdown.
UI Schema API
2. Fake Performance Data
Deterministic, content-type aware metrics
Four nullable columns on calendar: impressions, clicks, engagement_rate, comments. Generated during seed with content_type + platform aware ranges (case studies outperform polls on clicks, polls win on engagement). Consultant name hashed for per-person variation. Dashboard gets aggregate performance cards. One-click wipe sets all to NULL.
Schema API
3. Content Derivatives
Submission-level derivative tree
New derivatives table (FK to submissions). Each submission gets 2-4 fake derivatives based on topic category: bewijs gets 4 formats (linkedin-post, blog, email, infographic), contrair gets 2 (linkedin-post, newsletter). Inline tree view on topic cards. Nested JSON API endpoint for full topic → submission → derivatives tree.
Schema UI API
4. CSV + Google Sheets Export
Inline export panel on calendar
Date range picker defaulting to current month. CSV download (UTF-8 + BOM, comma-delimited, Content-Disposition header). Google Sheets sync via Apps Script webhook -- POST JSON to configured URL, ~20 lines of Apps Script. Webhook URL stored in new settings table. Button greyed out if not configured.
API UI

V2 Code Review

After the initial spec was written, the frontend was redesigned with a V2 dark theme (PP Monument Extended headings, DM Sans body, dark surfaces, red-to-orange gradient, trapezoid clip-paths). An Opus 4.7 agent reviewed the spec against the updated codebase and found 12 conflicts.

Critical (would break implementation)

#1Critical

CSV endpoint needs early exit before global Content-Type: application/json header. Added early route check pattern at top of api.php.

#2Critical

Soft delete is cross-cutting: ~15 SELECT queries need WHERE archived_at IS NULL. Spec now explicitly scopes the change.

#3Critical

One-to-many topic-to-calendar mapping (one topic posted on both LinkedIn and Dxfferent). Switched from JOIN to subquery with GROUP_CONCAT.

#4Critical

Schema migration for existing databases. Added PRAGMA table_info check + ALTER pattern instead of assuming DB delete.

#5Critical

Derivatives tree endpoint needs nested JSON, not flat rows. Added explicit PHP assembly logic and response shape to spec.

Important (would cause inconsistency)

#6Important

No V2 design tokens referenced. Added full token table: --surface, --line, --gradient, category jewel tones.

#7Important

Toggle button had no DOM insertion point. Specified: between .header and .stats in Board view.

#8Important

Editor toggleDone() didn't handle the new review status. Added explicit state machine for all transitions.

#9Important

DELETE /demo-data route conflicts with calendar DELETE handler. Changed to POST /demo-data/clear.

#10Important

Export panel needs structural DOM placement. Specified as full-width div below .cal-header.

Minor (good to clarify)

#11Minor

Auto-planner + kanban interaction confirmed working correctly. Auto-planned topics appear in Gepland column as expected.

#12Minor

Seed logic confirmed as api.php only. seed.json remains untouched.
