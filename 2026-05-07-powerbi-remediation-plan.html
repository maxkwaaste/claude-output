Stage 1

Strip Generic Markers from 510 Files

Scope: 510 files (427 STANDARD + 67 THIN + 10 SHELL + 6 CATEGORY) Type: Mechanical find-and-remove

Remove 10 known generic marker phrases from all affected files. These are filler sentences that the truthifier injected alongside real data but didn't clean up. The phrases are full sentences or sentence fragments that can be removed by deleting the containing paragraph or list item.

The 10 markers to strip:

  1. We pulled the numbers
  2. Here is what stands out
  3. where the gaps are
  4. what to do about it
  5. The data above paints a picture
  6. Look for patterns, outliers
  7. This report uses demo data
  8. Connect Proxuma Power BI to your own
  9. Schedule Recurring Review
  10. Connect Your Own Data

Expected outcome: All 872 files have generic_marker_count: 0 when re-audited.

Verification: Run python3 quality_audit_v2.py and confirm zero files have markers.

  • No dependencies. Can start immediately.

Stage 2

Build EN/NL Slug Mapping + Fix Hreflang on All 872 Files

Scope: All 872 files Type: Data mapping + mechanical rewrite

Every report file has two hreflang tags (EN and NL). Currently broken: each file uses its own slug for both languages. Example: EN file 28-day-backup-heatmap has NL alt URL /nl-powerbi/ai-gegenereerde-rapporten/28-day-backup-heatmap/ (wrong -- that's the EN slug in the NL path). It should point to /nl-powerbi/ai-gegenereerde-rapporten/28-dagen-backup-heatmap/.

The existing quality-audit-pairs-v2.json only matched 20 of 872 files. We need a better matching strategy to build the full EN/NL slug map.

Expected outcome: Every file's hreflang tags point to the correct counterpart URL. Files without a counterpart get x-default only (no broken cross-link).

  • No dependencies. Can run in parallel with Stage 1.
Session prompt for Stage 2 Context: Power BI report remediation Stage 2 of 10. Fix broken hreflang tags on all 872 report HTML files. ## The problem ~/ClaudeCode/powerbi-reports/ has 449 EN files in en/ and 423 NL files in nl/. Every file has hreflang link tags in the <head> pointing to EN and NL versions. The problem: each file uses its OWN slug for the alternate language URL. Example of the bug: - EN file slug: `28-day-backup-heatmap` - canonical: https://proxuma.io/powerbi/insights/28-day-backup-heatmap/ - NL alt (WRONG): https://proxuma.io/nl-powerbi/ai-gegenereerde-rapporten/28-day-backup-heatmap/ - NL alt (CORRECT): https://proxuma.io/nl-powerbi/ai-gegenereerde-rapporten/28-dagen-backup-heatmap/ - NL file slug: `28-dagen-backup-heatmap` - canonical: https://proxuma.io/nl-powerbi/ai-gegenereerde-rapporten/28-dagen-backup-heatmap/ - EN alt (WRONG): https://proxuma.io/powerbi/insights/28-dagen-backup-heatmap/ - EN alt (CORRECT): https://proxuma.io/powerbi/insights/28-day-backup-heatmap/ ## Step 1: Build the EN/NL slug mapping The existing file `quality-audit-pairs-v2.json` only matched 20 pairs by identical filename. That's insufficient. You need to build a better mapping. Read `quality-audit.json` (array of objects with fields: file, lang, canonical_url, alt_url, doc_title, html_title, etc.). Matching strategy (apply in order, stop at first match): 1. **Identical filename**: same filename in en/ and nl/ (20 pairs already found) 2. **Title match**: same doc_title in both languages (some reports have English titles in both) 3. **Slug similarity**: extract slug from canonical_url, use fuzzy matching (SequenceMatcher ratio > 0.6) between EN and NL slugs. Many NL slugs are Dutch translations of the EN slug. 4. **Content fingerprint**: for remaining unmatched, compute a fingerprint from the DAX queries in the file (DAX is language-independent). If two files share 2+ identical DAX blocks, they're pairs. Write a script `build-hreflang-map.py` that: 1. Loads quality-audit.json 2. Applies the 4 matching strategies in order 3. Outputs `hreflang-mapping.json` with structure: ```json { "pairs": [ {"en_file": "en/filename.html", "en_slug": "slug", "nl_file": "nl/filename.html", "nl_slug": "slug"}, ... ], "en_only": ["en/filename.html", ...], "nl_only": ["nl/filename.html", ...], "stats": {"paired": N, "en_only": N, "nl_only": N, "by_method": {"identical": N, "title": N, "fuzzy": N, "dax": N}} } ``` 4. Prints summary statistics Run the script. If fewer than 300 pairs are found, investigate the unmatched files and add more matching heuristics. ## Step 2: Fix hreflang tags in all files Write `fix-hreflang.py` that: 1. Loads `hreflang-mapping.json` 2. For each paired file: - EN file: update the NL hreflang URL to use the NL slug - NL file: update the EN hreflang URL to use the EN slug 3. For unpaired files (en_only, nl_only): - Remove the broken alternate hreflang link (the one pointing to a non-existent counterpart) - Keep the self-referencing hreflang and x-default 4. Also fix the alt_url in any JSON-LD structured data (same bug pattern) 5. Log all changes The hreflang tags look like: ```html <link rel="alternate" hreflang="en" href="https://proxuma.io/powerbi/insights/SLUG/" /> <link rel="alternate" hreflang="nl" href="https://proxuma.io/nl-powerbi/ai-gegenereerde-rapporten/SLUG/" /> ``` EN base path: `https://proxuma.io/powerbi/insights/` NL base path: `https://proxuma.io/nl-powerbi/ai-gegenereerde-rapporten/` ## Verification After running fix-hreflang.py: 1. Spot-check 5 paired files: open both EN and NL, confirm the hreflang tags cross-reference correctly 2. Spot-check 3 unpaired files: confirm they only have self-referencing hreflang 3. Run a validation script that for every file, checks if the alt URL slug actually exists as a file in the other language directory 4. Count: how many files were fixed, how many pairs established, how many remain unpaired ## After completion Rebuild staging: ```bash cd ~/ClaudeCode/powerbi-reports && python3 build-staging.py cd ~/ClaudeCode/powerbi-reports-staging && git add -A && git commit -m "stage 2: fix hreflang on all 872 files" && git push ``` Save `hreflang-mapping.json` -- it will be needed for gallery rebuilds in Stage 7.
