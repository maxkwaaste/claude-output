Completing Autotask write payloads

Three approaches for populating required fields per entity and handling per-tenant UDF variation. Evidence from the sandbox, trade-offs, a recommendation, and scope impact.

POST probe — one attempt, nothing created Sent POST /Projects with all 5 metadata-required fields plus the executor's standard UDF injection. Autotask returned: 
500 {"errors":["Error updating User Defined Field: Invalid User Defined Field: demo_filler_run_id."]}
Two findings. The 5 metadata-required fields were accepted (schema gate passed). But the UDF demo_filler_run_id does not exist in this sandbox, and the executor's _inject_udf adds it unconditionally. Every write would fail until the UDF is pre-created per tenant.

Open questions

  • Source of truth — runtime metadata probe, checked-in schema, or generated stubs?
  • Freshness — how do we detect and absorb Autotask schema changes?
  • Defaults — where do required-field values come from (plan, global config, inferred)?
  • Tenant variation — UDFs confirmed to vary. Required-fields variation unknown (probably minimal, not verified).

Approaches

Approach A — recommended Runtime probe + schema-driven defaults
Shape
On first seed call, GET /<Entity>/entityInformation/fields for all 7 write entities; cache to state/schema_cache/<entity>.json with a 24-hour TTL. Probe UDF existence the same way. Executor merges plan-provided values > global defaults > schema-driven defaults (first active picklist for enums, today for dates, today plus estimated hours for end dates). UDF injection becomes gated on probe result — if the UDF is missing, log a warning and skip.
Drift: HIGH Tenant flex: HIGH Impl: M Tests: M
Pros
Auto-adapts per tenant, zero schema drift in our code, handles UDF variation naturally, no human maintenance of a schema file.
Cons
~1-2s cold-cache latency on first seed. Seven extra reads per cache refresh (well inside the 120/min rate limit). Moderate test surface — need to mock the metadata endpoint and the cache invalidation path.
Approach B Checked-in schema + plan-level defaults
Shape
Commit core/schemas/<entity>.json, hand-curated from a one-time probe. Plan file gains an optional defaults block per entity. UDF list declared in the plan; validated via a single read-only metadata call at dry-run time.
Drift: LOW Tenant flex: M Impl: S Tests: S
Pros
Deterministic, fastest runtime, smallest test surface (JSON fixtures), no runtime API coupling for the main write path.
Cons
Schema drift is a human problem — someone has to notice and update. Per-tenant UDF still needs a runtime probe (partial overlap with Approach A). Plan files grow heavier with every required-field recipe.
Approach C Generated stubs from OpenAPI
Shape
Pull or build an Autotask OpenAPI spec, codegen Python dataclasses per entity with required-field validation, regenerate on a schedule.
Drift: M Tenant flex: LOW Impl: L Tests: L
Pros
Strongest typing, editor autocomplete, catches drift at codegen time.
Cons
Autotask does not publish a clean OpenAPI spec — the same metadata endpoints feed it, so we re-derive Approach A's data anyway, plus a codegen pipeline. Heavy for a 7-entity internal tool. Per-tenant UDF still needs a runtime probe.

Recommendation

Approach A — runtime metadata probe with cached schema

It is the only approach that naturally handles the per-tenant UDF variation already confirmed on sandbox dxfferentSB05122022.nl. Schema drift is absorbed without human intervention, and the scope stays inside this feature branch. The extra runtime cost is one bounded burst of seven reads on a 24-hour cycle, inside the 120-per-minute rate budget.

Scope and branch fit

Approach A is one feature branch. Touches core/ (new schema cache module, small additions to autotask_client), executor/ (defaults merge in each _create_*), and adds state/schema_cache/ to .gitignore. No refactor of the retry logic that just shipped. Stay on feat/mcp-server.

Decision point Approve Approach A, pick a different one, or push back on the evidence. Once approved, next steps are the design doc at docs/superpowers/specs/2026-04-21-executor-payload-completion-design.md, then the implementation plan via the writing-plans skill. Neither will touch executor code this session.

Postscript — decision override (2026-04-21)

Max rejected the recommendation (Approach A) and chose a narrowed form of Approach B. Framed as a safety override. Original trade-offs above are preserved for audit — they still reflect the tooling cost math, which did not drive the decision.

Decisive argument

Safety against unintended Autotask state Approach A's schema-driven defaults quietly pick the first active picklist value when a required enum is missing. For status that value can be any member of an 8-member or 17-member list on any given tenant — including "Complete". Seeding a demo whose projects and tickets are all marked complete on arrival is a customer-visible failure mode we cannot detect from our side. The safety model for this tool is no surprise writes. Silent enum defaults violate it. Cost-of-tooling trade-offs were not the tiebreaker.

Chosen shape — narrowed Approach B

The safety contract is no inference, at any layer. Values come from explicit sources or the run halts:

  1. Checked-in schema fixtures at core/schemas/<Entity>.json, built from the /tmp/autotask-metadata/ probe. Each fixture records required field names, types, and the picklist value set captured at probe time. Pure data, no defaults.
  2. Plan file extension — optional per-entity defaults block. Defaults are explicit literals written by a human, never inferred from schema.
  3. Runtime UDF existence probe — one read-only GET per dry-run per entity targeted with UDF injection, against /<Entity>/entityInformation/userDefinedFields. If demo_filler_run_id is missing on the tenant, dry-run HALTS with a clear message. No silent skip — Max decides whether to create the UDF or adjust the plan.
  4. Executor value resolution — strict, no fallthrough: plan record value → plan entity-level default → HALT. No schema-driven default layer exists.
  5. Validator enforces the same order at dry-run, so seed never encounters a plan that would halt mid-write.

What was kept from A, what was rejected

From Approach AFateWhy
Runtime UDF existence checkKept, narrowedUDF variation is already confirmed per-tenant. One dry-run probe per entity is cheap and the only way to detect the gap observed on dxfferentSB05122022.nl.

Open questions surfaced by the override

  • Fixture refresh trigger — picklist values may drift on any tenant at any time. Do we version the fixture and require a manual re-probe on some cadence, fail loudly when Autotask accepts an integer status our fixture does not know about, or both?
  • Per-tenant fixture separation — the current fixtures are captured from one sandbox. If a second tenant is onboarded with a different picklist set, do we keep one fixture and union the value sets, or fork per tenant?
