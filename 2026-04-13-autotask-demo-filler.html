Demo Filler: Realistic MSP Data Seeder

One-shot seeder that fills the Proxuma Autotask demo environment with realistic projects, tickets, service calls, and time entries. Four independent safety gates prevent touching developer data. Built in a single session using subagent-driven development.

How It Works

The system splits into two phases: plan generation (Claude creates a JSON plan describing every record to seed) and execution (Python applies the plan via the Autotask REST API on M1 Pro). The plan is human-reviewed before any writes happen.

Safety Architecture

Four independent gates plus a kill switch. Each one stops the run if triggered. They compose in the HTTP client so every write passes through all of them.

Gate 1
Credential Isolation
Dedicated API user + tracking ID. Own 10k/hr rate-limit bucket. Credentials at ~/.autotask/demo.env, never in repo.
Gate 2
Resource + Account Guard
Every write payload is inspected before leaving the process. Resource and account IDs must be in the allowed set or the call is blocked.
Gate 3
UDF Tag on Every Record
demo_filler_run_id = <uuid> injected automatically. Developers can spot demo-filler records at a glance in the Autotask UI.
Gate 4
Manifest-Gated Wipe
Wipe only deletes IDs from its own manifest. Never queries "all records with UDF X". Survives edited or hand-created records.
Kill Switch
PAUSE File
touch ~/.autotask/PAUSE from any terminal stops all writes instantly. Circuit breaker also halts after 5 consecutive failures per endpoint.
Rate Limit
Token Bucket
120 writes/min (30% safety margin on Autotask's 10k/hr budget). 2 concurrent threads per endpoint. Adaptive backoff on 429/5xx.

Execution Flow

Records are created in dependency order. The manifest is written per-record (not batched), so a crash mid-run loses zero progress. Resume picks up exactly where it stopped.

What Happens Next

Before the First Live Run

  • Create dedicated API user + tracking ID in Autotask demo environment
  • Create UDF demo_filler_run_id (string, 36 chars) on Projects, Tickets, ServiceCalls, Tasks, TimeEntries
  • Write credentials to ~/.autotask/demo.env on M1 Pro
  • Run template reader with master template project ID
  • Update tests/smoke_plan.json with real resource + account IDs
  • Run smoke test: seed, verify, wipe, verify
  • Run demo-vision interview, then generate real seed_plan.json
