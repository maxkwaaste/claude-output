Replacing 2,500 Fabricated DAX Queries with Real Ones

Design for validating, rewriting, and injecting real DAX queries into 872 Power BI report pages, then updating report content with actual data from the Proxuma Demo dataset.

The Problem

Every DAX query embedded in the 872 report pages was generated by AI without consulting the actual Power BI schema. Table names often coincidentally match real tables, but column names, measure references, and query logic are invented. None of the ~3,500 queries execute successfully.

The reports also display specific numbers (KPIs, tables, narratives) that were generated alongside the fake DAX. Once we have real queries producing real results, the displayed data needs to reflect those results.

Constraint Visual design is frozen. The page layout, CSS, morph bar, search hero, pipeline section, teal report frame, section numbering, FAQ accordion, CTA block, and JSON-LD schema must not change. Only the DAX code blocks and data-driven content (KPI values, table rows, narrative paragraphs) get updated.

Approach: Query Library First, Then Inject

Rather than processing 872 files independently, we build a validated DAX query library organized by report topic, then inject results into both the EN and NL versions of each report. This deduplicates across language pairs and creates a reusable asset for future report generation.

Phase 1: Schema Snapshot

Pull the complete schema from the Proxuma Demo dataset and cache it locally as the single source of truth for DAX authoring. This prevents hallucination at the root: if a table or column isn't in the snapshot, it doesn't exist.

This snapshot is taken once and pinned for the entire pipeline run. If the schema changes mid-run, the snapshot is stale and the run must restart.

Phase 2: Extract and Pair

2a. Re-extract DAX from all files

The existing dax-extracted-queries.json only covers 347 files (from the April 3 audit). We need fresh extraction from all 872 files in en/ and nl/ directories.

For each file, extract:

  • Filename and language (en/nl)
  • Each DAX query (from <pre><code> inside .dax-toggle)
  • The section title each DAX belongs to (from the parent .doc-section-title)
  • KPI labels and current values
  • Table column headers
  • Report title and subtitle (the "question" the report answers)

Output: dax-extraction-full.json with one entry per file, each containing an array of query objects with their surrounding context.

2b. Build EN/NL topic pairs

The existing question-file-mapping.json covers only 124 topics (99 EN, 90 NL). There are 449 EN and 423 NL files. The remaining ~350 pairs need to be matched.

Pairing strategy:

  1. Start with the 124 known pairs from the mapping file
  2. For unmatched files: compare report titles semantically (the EN title "SLA Performance by Priority" maps to the NL title "SLA Prestaties per Prioriteit")
  3. Where DAX queries are identical between an EN and NL file, that's a confirmed pair
  4. Remaining orphans (EN without NL or vice versa) get flagged for manual review

Output: topic-pairs.json with ~450 entries, each containing the EN filename, NL filename (if exists), report topic, category, and the number of DAX sections.

Phase 3: DAX Authoring (Opus-Supervised)

This is the core of the pipeline. For each of the ~450 unique report topics, an agent writes valid DAX queries, executes them, and stores the results.

Architecture

RoleModelResponsibility
SupervisorOpusReviews every batch of 20 query sets. Checks for hallucinated columns/tables, nonsensical results, and queries that "pass" but return empty or meaningless data. Veto power over the batch.
WorkerSonnetReads the schema snapshot + report context (section titles, KPI labels). Writes DAX queries. Executes via mcp__powerbi__execute_dax. Retries up to 3 times per query on failure.

Anti-hallucination rules (enforced in worker system prompt)

  1. Schema-only columns: Every table and column reference in DAX must exist in schema-snapshot.json. No exceptions.
  2. Measure verification: Every measure reference (e.g. [CSAT - Average Rating]) must match a measure name from the snapshot exactly.
  3. Execute-or-reject: A query that doesn't execute successfully against the real dataset is rejected. No "should work" assumptions.
  4. Non-empty results: A query that returns zero rows or all nulls is flagged for review (it might be valid but needs a different approach).
  5. No invented relationships: If a query needs to join tables, the relationship must exist in the schema or the query must use explicit TREATAS/CROSSFILTER.
  6. Schema provided as literal context: The worker receives the relevant table schemas inline, not just the snapshot path. No room for "I think this column exists."

Worker flow per topic

1. Read report context (title, section titles, KPI labels, table headers)
2. Identify which tables/columns are relevant from schema snapshot
3. For each section that needs a DAX query:
   a. Write EVALUATE query using only verified schema elements
   b. Execute via mcp__powerbi__execute_dax
   c. If fails: read error, fix query, retry (max 3)
   d. If passes: store query + result set
   e. If 3 failures: flag topic for Opus review
4. Package: { topic, queries[], results[], status }

Language handling

EN and NL files for the same topic share the same DAX queries and the same result data. The difference is:

  • DAX code blocks: Identical in both (DAX is DAX, no translation)
  • KPI values: Identical (numbers are numbers)
  • Table data: Identical
  • Narrative text: Written in the target language, referencing the same numbers
  • Section titles: Already in the correct language, not changed

Phase 5: Visual QA on M1 Pro

Runs on the M1 Pro server over Ethernet. Every 10th report gets a full screenshot comparison.

QA pipeline

  1. Pre-screenshot: Before injection, render each report in a headless browser and save a baseline screenshot
  2. Post-screenshot: After injection, render again and save comparison screenshot
  3. Diff check: Compare screenshots. Flag if structural layout changed (element positions, missing sections, broken rendering). Ignore expected changes (different numbers, different text content).
  4. HTML validation: Check that every file has valid HTML (no unclosed divs, no broken DAX toggles)
  5. DAX toggle test: Verify that every .dax-toggle still opens/closes correctly
Frequency Full screenshot comparison on every 10th file. HTML validation and DAX toggle check on every file. If a screenshot diff flags a problem, pause and investigate before continuing.

Rate limiting

Power BI API has rate limits. The pipeline should:

  • Space DAX executions at least 1 second apart
  • Back off on 429 responses with exponential delay
  • Log every execution with timestamp, query, result status
  • Resume from where it left off if interrupted (checkpoint after each topic)

Output Artifacts

  • dax-extraction-full.json — all queries + context from all 872 files
  • topic-pairs.json — EN/NL pairing with ~450 topics
  • dax-library.json — validated query library: topic → queries → results
  • injection-log.json — per-file log: what changed, before/after hashes

  • qa-failures.json — flagged files needing manual review

Bottom Line

872 reports, ~3,500 fake DAX queries, zero of them work. The fix is a 5-phase pipeline: snapshot the real schema, extract and deduplicate across EN/NL pairs, write valid DAX under Opus supervision with strict anti-hallucination rules, inject real results into the existing HTML shell, and QA with screenshot diffs. Runs unattended on M1 Pro, takes 8-15 hours. The 30 "other" files are never touched.
