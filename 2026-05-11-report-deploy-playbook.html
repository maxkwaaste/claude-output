Infrastructure

Step 1: Extract CSS Template

Extracts the canonical CSS from a known-good report file and locks it as the single source of truth. Every future report and batch fix references this file.

Worker Step 1 ~5 min
TASK: Extract the locked CSS template from a known-good report. SOURCE: ~/ClaudeCode/powerbi-reports/en/proxuma-io-sla-performance-post.html OUTPUT: ~/ClaudeCode/powerbi-reports/templates/teal-frame-css.html WHAT TO DO: 1. Read the source report file. 2. Find all CSS inside the second <style> block (the one after the morph bar CSS, starting with the comment about "LOCKED CSS TEMPLATE"). This is the teal-frame + report-specific CSS. If there's no comment marker, it's the style block containing `.prx-report` rules. 3. Extract that entire style block into templates/teal-frame-css.html. Include the <style> tags. 4. Add a comment at the top: <!-- LOCKED CSS TEMPLATE - Do not modify. Extracted from sla-performance on YYYY-MM-DD. Any report-specific CSS goes in a SEPARATE style block. --> 5. Verify the extracted CSS contains these key selectors (grep for them): .prx-report, .report-frame-outer, .doc, .doc-header, .doc-kpi-row, .doc-kpi, .doc-section, .dax-toggle 6. Count the CSS rules. Report the count. DO NOT modify any CSS values. Copy verbatim. Do not "clean up" or reformat. Create the templates/ directory if it doesn't exist. When done: git add and commit to the powerbi-reports repo with message "chore: extract locked CSS template from sla-performance"

Step 3: Build Visual Audit Script

Creates an automated validation script that scans all 872 files for the six error classes identified in the analysis. This runs before every deploy.

Worker Step 3 ~15 min
TASK: Build visual-audit.py that scans all Power BI report HTML files for visual errors. OUTPUT: ~/ClaudeCode/powerbi-reports/visual-audit.py REFERENCE: Read ~/ClaudeCode/powerbi-reports/qa-validate.py for the existing file I/O patterns and HTML parsing approach used in this repo. THE 6 CHECKS (implement all): 1. DIV BALANCE: Parse each HTML file, count opening and closing div tags. Flag files where counts don't match. Report the delta. 2. CLASS WHITELIST: Grep inside .prx-report content for these drift markers. Flag any file containing them: - .bar-chart, .bar-row, .bar-fill (should be .patch-row, .patch-bar-wrapper, .patch-segment) - .kpi-grid, .kpi-card, .kpi-value (should be .doc-kpi-row, .doc-kpi, .doc-kpi-value) - .section-card (should be unwrapped into .doc-section) - .inline-dax (should be .dax-toggle) - .metric-card, .stat-card (should be .doc-kpi) - .data-table, .stats-table (should be plain table in .doc-body) 3. COLOR AUDIT: Extract all hex color values from inline style attributes and style blocks. Flag non-standard colors. Allowed palette: #0f766e, #115e58, #134e4a, #0f2240 (teals/navy) #1B365D, #1b365d (document navy) #2c3e50 (text) #333, #333333 (text alt) #fafbfc, #f1f5f9, #e2e8f0 (grays) #64748b, #94a3b8 (muted text) #10B981, #ef4444, #f59e0b (status green/red/amber) #5eead4 (morph bar link teal) #fff, #ffffff, #000, #000000 (black/white) Plus any colors inside SVG elements (skip those) and standard badge colors (DBEAFE, FDE68A, FECACA, E9D5FF, A7F3D0, FEF3C7 etc.) 4. FONT AUDIT: Grep for font-family declarations. Flag any containing: Inter, Montserrat, Roboto, Arial, Helvetica. Allowed: Open Sans, DM Serif Display, JetBrains Mono, Playfair Display, system fonts (-apple-system, BlinkMacSystemFont, sans-serif, serif, monospace). 5. ARTIFACT SCAN: Grep for known filler phrases: "We pulled the numbers", "Schedule Recurring Review", "Let's break down", "TODO", "PLACEHOLDER", "Lorem ipsum" Also flag files with more than one FAQPage JSON-LD block. 6. STRUCTURE CHECK: Verify each file has: - Exactly one .prx-morph-bar - Exactly one .report-frame-outer - Exactly one .doc element - At least one .doc-kpi-row - At least one .doc-section OUTPUT FORMAT: Print a summary table showing pass/fail counts per check. Then list every flagged file grouped by check type, with the specific issue found. At the end, write results to ~/ClaudeCode/powerbi-reports/AUDIT-RESULTS.md. SCAN TARGETS: All .html files in en/ and nl/ directories. Skip GOLDEN-STANDARD.html, gallery files, index.html, and anything in templates/. Run the script after building it. Report results. When done: git add visual-audit.py and AUDIT-RESULTS.md, commit with "feat: add visual audit script with 6-check validation"

Step 4: Fix Audit Findings

Batch-fixes whatever the audit found. The prompt adapts based on actual findings.

Worker Step 4 ~20 min (depends on findings)
TASK: Fix all visual audit findings in the Power BI report library. READ FIRST: - ~/ClaudeCode/powerbi-reports/AUDIT-RESULTS.md (the audit output from step 3) - ~/ClaudeCode/powerbi-reports/templates/teal-frame-css.html (the canonical CSS) RULES: 1. Only fix what the audit flagged. Do not "improve" other things. 2. For CSS class drift: replace wrong classes with correct ones using this mapping: .bar-chart/.bar-row/.bar-fill -> .patch-row/.patch-bar-wrapper/.patch-segment .kpi-grid/.kpi-card/.kpi-value -> .doc-kpi-row/.doc-kpi/.doc-kpi-value .section-card -> unwrap into .doc-section .inline-dax -> .dax-toggle (note: different HTML structure, check each case) .metric-card/.stat-card -> .doc-kpi .data-table/.stats-table -> remove class, leave plain table 3. For div imbalance: find the specific broken nesting and fix minimally. Do not restructure. 4. For wrong colors: replace with nearest palette color. Do not guess, use the allowed palette from the audit script. 5. For wrong fonts: replace with Open Sans (body), DM Serif Display (titles), or JetBrains Mono (code). 6. For content artifacts: remove filler phrases. Do NOT inject replacement text. Just delete. 7. For duplicate FAQ JSON-LD: keep the first one, remove extras. NEVER touch: - JavaScript - DAX query content inside code blocks - Morph bar CSS (first style block) - Proxuma SVG logo - JSON-LD structured data (except duplicate FAQPage cleanup) After all fixes, re-run the audit: python3 ~/ClaudeCode/powerbi-reports/visual-audit.py TARGET: Zero findings on re-run. If some findings can't be fixed automatically (need manual review), list them in AUDIT-RESULTS.md under a "Manual Review Needed" heading. When done: git add all changed files. Commit with "fix: resolve visual audit findings (N files fixed)" where N is the count.
Infrastructure

Step 5: Build Deploy Library

Creates the deployment script that pushes report HTML to WordPress pages via SSH.

Worker Step 5 ~15 min
TASK: Build deploy_reports_lib.py for pushing Power BI report HTML files to WordPress via SSH. OUTPUT: ~/ClaudeCode/powerbi-reports/deploy_reports_lib.py REQUIREMENTS: 1. PROTECTED_PAGE_IDS = {3299, 3495, 3401, 5655, 3389, 4887} Every function that touches a page ID must check against this set first. Hard abort with error message if a protected page is targeted. 2. Core functions: - get_wp_pages(ssh_config, parent_id) -> list of {id, slug, title} Runs: wp post list --post_parent={parent_id} --post_type=page --fields=ID,post_name,post_title --format=json - build_slug_map(en_pages, nl_pages, report_files) -> dict mapping filename to page_id Match report filenames to WP page slugs. EN parent: 3299, NL parent: 3495. - push_report(ssh_config, page_id, html_content, dry_run=False) Uploads HTML via temp file on server, then runs PHP: $wpdb->update('wp_posts', ['post_content' => file_get_contents('/tmp/report_{page_id}.html')], ['ID' => {page_id}]); In dry_run mode: only upload the temp file and print what WOULD be updated. Do not run the PHP. - batch_deploy(ssh_config, slug_map, report_dir, dry_run=False, batch_size=50, delay=5) Deploys in batches with delay between batches. Logs progress to stdout. Returns: {success: [...], failed: [...], skipped: [...]} - flush_cache(ssh_config) Runs: cd ~/www/proxuma.io/public_html && wp cache flush && wp sg purge 3. SSH config dict format: {"host": "es33.siteground.eu", "port": 18765, "user": "u1434-nm1gftpe9yl0", "key": "~/.ssh/id_ed25519"} Support overriding for staging via a --staging flag or separate config. 4. CLI interface: python3 deploy_reports_lib.py --dry-run # dry run all python3 deploy_reports_lib.py --dry-run --limit 10 # dry run first 10 python3 deploy_reports_lib.py --deploy --limit 10 # deploy first 10 python3 deploy_reports_lib.py --deploy # deploy all python3 deploy_reports_lib.py --staging --deploy # deploy to staging env Add --en-only and --nl-only flags. 5. HTML extraction: Read each file, extract content from inside the body tag. The reports are full HTML documents but WordPress only needs the body content (everything between body tags, excluding the tags themselves). 6. Logging: Print progress as "Deploying [N/total] slug -> page_id ... OK/FAIL". Write a deploy log to DEPLOY-LOG.md with timestamp, files deployed, success/fail counts. TEST: Run with --dry-run --limit 1 to verify it connects via SSH and reads the WP page list. Report the output. When done: git add and commit with "feat: add deploy_reports_lib.py with protected page safeguards"

Phase 2: Staging Setup

One manual step, then the first real deploy.

Manual

Step 6: Set Up SiteGround Staging

This is the one step you do yourself in the browser. No Claude session needed.

Manual step
  1. Log into SiteGround Site Tools for proxuma.io
  2. Go to WordPress > Staging
  3. Click Create Staging Copy
  4. Wait for the clone to complete (a few minutes for the full site)
  5. Note the staging URL and staging SSH details that SiteGround gives you
  6. Go back to the conductor session and tell it: "Step 6 done. Staging URL is [URL]. Staging SSH is [details if different from prod]."

The conductor will update the deploy script config for the staging environment.

Step 9: Fix QA Findings

Fixes the blocking issues from the QA pass. Adapts based on what was actually found.

Worker Step 9 ~20 min (depends on findings)
TASK: Fix all BLOCKING visual issues found during QA of the staging test batch. READ FIRST: - ~/ClaudeCode/powerbi-reports/QA-FINDINGS.md (the issues to fix) - ~/ClaudeCode/powerbi-reports/templates/teal-frame-css.html (canonical CSS) - ~/ClaudeCode/powerbi-reports/GOLDEN-STANDARD.html (visual reference) RULES: 1. Fix BLOCKING issues only. Leave COSMETIC for a later pass. 2. If the fix affects CSS that's shared across all 872 files (e.g., a color value in the inline CSS), you need to apply it to ALL files, not just the 10 test files. Build a batch fix script if needed. 3. For WP theme CSS conflicts: the fix goes into the report CSS as higher-specificity overrides. Add them to the template CSS (teal-frame-css.html) first, then propagate to all files. Example: if the WP theme overrides h2 styling, add .prx-report .doc h2 { ... !important } to the template. 4. When changing the template CSS, update it in teal-frame-css.html AND in every report file's inline CSS. Write a Python script (fix-css-propagate.py) to do this. 5. After fixes, re-run visual-audit.py to make sure fixes didn't introduce new issues. 6. Re-deploy the 10 test files to staging to verify the fixes. NEVER: - Rewrite the entire CSS. Only add targeted overrides for specific conflicts. - Change frozen zone values (colors, paddings) unless QA-FINDINGS.md specifically says those values are wrong. - Touch content (text, DAX queries, tables data). When done: commit all changes with "fix: resolve blocking QA findings from staging test batch"

After Step 10

Go back to the conductor session. It will review PRODUCTION-READY.md and tell you whether to proceed with production deploy. The production deploy is the same command without --staging. After production deploy: flush cache, verify 5 live pages, update gallery ACF on pages 3299 and 3495 if needed.

Rollback plan If production deploy goes wrong, all original page content is in WordPress revision history. Each page's last revision before the deploy is the rollback point. The deploy script should log the pre-deploy content hash for each page so you can verify rollback if needed.
