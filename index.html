What is this?

The NeXum Claude Foundation is a shared configuration system so everyone on the team works with Claude Code in a consistent, quality-controlled way. It installs rules, hooks, skills, commands, and agents into your ~/.claude/ directory.

It's optional for you. When you work in ~/nexum/ projects, the foundation workflow applies. For Proxuma, Power BI, and personal projects, your normal setup continues.

The foundation lives at github.com/NeXum-co/claude-foundation. Installed on both M4 Air and M1 Pro.

Three Layers

L1 Foundation

Team-wide

Universal rules, hooks, skills, commands. Shared by everyone. Source: claude-foundation repo. Installed via install.sh.

L2 Project

Project-specific

Frontend standards, backend architecture, deploy rules. Auto-loaded from each project's .claude/ directory when you cd into it.

L3 Personal

Your setup

Your own preferences, experiments, and memory in ~/.claude/. Not shared. This is where your Proxuma skills, Power BI commands, and personal hooks live.

Direction matters: upward via PR (proven, reviewed), downward via pull/install (available, not mandatory). If something works well personally (L3), you can propose it to the project (L2) or the whole team (L1).

Model Strategy

The foundation uses a deliberate model hierarchy. Each model has different strengths.

RoleModelWhen
Orchestrator Opus 4.6 Your main session. Planning, brainstorming, reviewing, verifying. High recall (MRCR 78%), good at inference and synthesis.
Implementation Opus 4.7 Building and fixing code. One agent per phase from a prepared plan. Best at literal instruction-following (SWE-bench 87.6%), but low recall.
Research Sonnet Code search, investigation, fact-finding via Explore agents. Fast and cheap for read-only work.

The rule: As long as there are open questions, use 4.6. Once the plan is done, hand off to 4.7. Opus 4.6 always verifies what agents produce. Agent output is never trusted blindly.

Development Workflow

For features and larger changes (>200 lines of code), the workflow is:

Brainstorm → Plan → Gate 1 → Execute (4.7) → Gate 2 → Verify + PR
  • Gate 1 (Plan Quality): Is each step a vertical slice? Are assumptions verified? Does it fit one session?
  • Gate 2 (Output Quality): Is every planned item actually done? Tests run with evidence? No debug clutter?

For complex multi-step tasks, use /crispy to structure the work into CRISPY stages (Context, Research, Ideate, Structure, Plan, Yield, PR).

For small tasks (bug fixes, quick changes), just describe what you need. The loaded rules guide Claude automatically.

Git Workflow

  • Always work on a feature branch (feature/, fix/, docs/, chore/ prefix). Never commit to main.
  • Open a PR and get at least 1 review before merging. Squash-merge.
  • Use conventional commits: feat:, fix:, docs:, refactor:
  • New features require tests. Bug fixes require a regression test.

Slash Commands

Type these in Claude Code. They're installed at ~/.claude/commands/.

/crispy skill

Research-to-PR methodology in 7 stages (Context, Research, Ideate, Structure, Plan, Yield, PR). Use for features >200 LOC or when agent quality degrades from instruction overload. Each stage gets its own prompt with <40 instructions.

/test skill

Test websites, APIs, and UI. Routes to Chrome DevTools MCP (interactive debugging), Playwright (headless E2E), PHPUnit (backend), or curl (API) based on what you ask. "Test dit in de browser" goes to Chrome DevTools, "draai de tests" goes to Playwright CLI.

/jira skill

Create, update, or query Jira issues. Enforces ticket writing standards scaled to complexity, correct issue type hierarchy (Improvement > Epic > Task, Bug standalone), and consistent formatting. All Jira content in English.

/externe-teksten skill

Write external-facing Dutch text: landing pages, sales copy, workshop descriptions. Enforces Voice of Customer, PAS structure, anti-AI-slop checklist. Includes forbidden words list and B2B IT-specific patterns to avoid.

/video-summarizer skill

Summarize video content with structured output. Extracts key topics, timestamps, and takeaways.

/deploy command

Deploy projects to production. Auto-detects deploy method per project from a registry. /deploy nexum deploys all NeXum components via SSH. /deploy nova deploys via git push. Supports --dry-run.

/transcribe command

Transcribe a screen recording (OBS, etc.) and generate a structured report with screenshots at key moments. Auto-finds most recent OBS recording if no path given.

/launch-local command

Start local development environments. /launch-local proxuma starts the Proxuma stack, /launch-local investment-portal starts the IP stack. /launch-local stop stops everything.

/harness-list command

Browse all available items in the shared Claude Code harness catalog. Filter by --type (skill, hook, rule, command) or --tag.

/harness-pull command

Install a specific item from the shared harness. Format: /harness-pull type/name (e.g. /harness-pull skill/proxuma-i18n).

/harness-status command

Compare your local install with the latest foundation. Shows what's pulled from the shared harness, what's personal, and what has updates available.

Rules (10)

Rules load automatically every session. They're behavioral constraints, not skills you invoke. Installed at ~/.claude/rules/.

code-quality rule

Scope discipline: no features, refactoring, or improvements outside the requested task. A bugfix doesn't need surrounding cleanup. Keep changes minimal and focused.

compliance-zones rule

3-zone data model for AI usage. Green (free), Orange (careful), Red (never). Determines what data can go through which AI tools. Always active.

crispy-principles rule

4 discipline principles on top of superpowers: research isolation, instruction budget (<40 per prompt), vertical slices (not horizontal layers), artifacts over context (write to files).

git-workflow rule

Branch from main, use feature/fix/docs/chore prefixes, conventional commits, PRs with review, squash-merge, delete after merge.

mcp-usage rule

Rules for using MCP servers. They process data locally but output goes to context window. Compliance zones apply to MCP tool output too.

model-delegation rule

Opus 4.6 orchestrates, 4.7 implements, Sonnet explores. Never implement complex plans directly in orchestrator. Always verify agent output.

output-quality rule

Quality checks between execution and finish (Gate 2). Every planned item actually implemented? Tests run with evidence? No console.logs, TODOs, or debug code?

plan-quality rule

Quality checks between plan and execution (Gate 1). Each step a vertical slice with concrete verification? Assumptions checked? Fits one session?

security rule

Never commit .env, credentials, API keys. Use .env.example as template. Ask when uncertain if file contains sensitive data.

testing rule

New features need tests before merge. Bug fixes need regression tests. Refactoring must keep existing tests passing. Config changes need smoke tests.

Hooks (7)

Hooks run automatically before/after tool calls. They're JS scripts in ~/.claude/hooks/. No invocation needed.

post-edit-format.js

Auto-formats JS/TS files after every Edit/Write. Detects Biome or Prettier by looking for config files. Skips silently if no formatter found (safe for PHP, Python, etc.).

post-edit-typecheck.js

Runs TypeScript check after editing .ts/.tsx files. Finds nearest tsconfig.json, runs tsc --noEmit, reports errors for the edited file only. Skips non-TS files.

console-warn.js

Warns about console.log statements after edits to JS/TS files. Reports line numbers so you can remove debug statements before committing.

git-push-reminder.js

Prints a reminder before git push to make sure changes are reviewed and tests pass. Non-blocking.

cost-tracker.js

Tracks token usage and estimated cost per session. Appends JSONL to ~/.claude/metrics/costs.jsonl. Runs at session end.

session-summary.js

Writes a session summary (tasks, files modified, tools used) to ~/.claude/sessions/. Next session can load the most recent summary for context.

harness-staleness-check.js

Checks at session start whether your foundation install is outdated (>7 days since last pull). Prints a one-line warning if so.
