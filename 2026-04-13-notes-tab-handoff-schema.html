Claude Coding Workflow: Notes Tab + Handoff Schema Upgrade

Built two improvements to Max's workflow: a notes-tab helper for iTerm tab titles and clipboard handoff prompts, and an extended CLAUDE-DATA schema so successor Claude sessions can resume work from the HTML alone.

Problem

Cleared Claude sessions lose context even with MEMORY.md and pushed output-page HTMLs. The CLAUDE-DATA block lacked structure for machine state, artifacts, external changes, and decisions. Tab titles in iTerm stayed generic, giving Max no visual cue about what each session was doing. And there was no clipboard-ready handoff prompt for the next session.

Notes-tab system

A helper script at ~/.claude/helpers/tab-status.sh takes two arguments: a caveman-style tab title and a handoff prompt. It sets the iTerm tab title via OSC 1 escape codes and pipes the handoff prompt to pbcopy. A rule in global ~/.claude/CLAUDE.md under "Notes Tab (ALWAYS ACTIVE)" tells every Claude session to call the helper once the task is clear and again before /clear.

Key detail The helper is POSIX bash, 50 lines, chmod +x, and swallows /dev/tty errors so it works inside agent sub-shells too.

Schema extension

Five new required top-level fields were added to the CLAUDE-DATA JSON block:

FieldPurpose
session_statePhase, status, progress, work done, blockers, uncommitted changes, next action type
artifact_pointersMemory files (with verified_live), git branch, repo path, temp files, github links, WP IDs, commit SHA
external_stateWP cache, drafts pending (with execution_method), screenshots, other out-of-repo changes
project_contextProtected resources, conventions applied (memory files consolidated into artifact_pointers only)
decision_logNon-obvious choices with rationale, rejected alternatives, trade-offs, contingencies (one level only)

Pre-push gate

The 5-question handoff test was updated to reference the new fields explicitly. Each question now names the specific sub-fields it gates on, including next_action_required, verified_live on memory files, execution_method on drafts, and the one-level contingency cap.

Five stress-tested tweaks

  1. next_action_required enum on session_state: none | user-exec | user-decision | tool-waiting | claude-continue. Names what type of action unblocks the next step.
  2. artifact_pointers subfields all optional with "fill what applies" rule. Not every category applies to every file type.
  3. execution_method enum on drafts_pending: slack | email | github-issue | conversation | wp-admin | other.
  4. Structured memory_files objects with name, version_date, verified_live. Single source of truth in artifact_pointers (removed from project_context).
  5. Optional contingencies on decision_log capped at one level. No nested contingencies; if a branch needs its own decisions, create a new top-level decision_log entry.

Three follow-up patches (this session)

  1. Pre-push gate wording: replaced generic questions with field-specific references (session_state sub-fields, verified_live, execution_method, contingencies cap).
  2. Memory-files dedup: removed memory_references from project_context schema and example. Added note that memory files live only in artifact_pointers.memory_files[].
  3. Contingencies cap: added explicit one-level rule to the decision_log schema description.
