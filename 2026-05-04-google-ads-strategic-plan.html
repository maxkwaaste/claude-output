1. Research Findings

Product Features That Drive Ad Copy

Proxuma solves one core problem: Autotask's native project management is broken. From 140 demo transcripts, these features consistently move prospects from interested to trial:

  1. Visual project timeline / Gantt with drag-and-drop task management
  2. Dispatch calendar where dragging a ticket onto an engineer creates a service call + Outlook invite in 2 clicks
  3. Bulk editing of 50+ tasks at once (dates, assignees, statuses)
  4. Capacity planning with real-time color-coded resource availability
  5. Own Outlook integration that works every time (replaces Autotask's broken Exchange sync)
  6. Shift planning for NOC/SOC rotations
  7. Multi-project overview showing all projects on one screen

Engineers never need to log into Proxuma. Only planners and PMs use it. Everything syncs back to Autotask.

Why It Worked

The golden period succeeded because it targeted people already searching for Autotask-specific solutions. "Autotask project management" is a high-intent keyword: the searcher already uses Autotask and already knows the project module is broken. John's campaigns failed because "MSP planning tools" attracts people who may not even use Autotask. The lesson: stay in the Autotask ecosystem.

Geographic Targeting

Recommendation Start with USA + UK + Australia only. These are English-speaking markets where Proxuma has proven traction. The golden period data came primarily from US traffic. Netherlands/Belgium prospects find Proxuma through Dutch channels (LinkedIn, referrals, Dxfferent), not Google Ads in English. If budget allows expansion after month 2, add Canada and New Zealand. Do NOT create separate campaigns per geography at this budget level. One campaign, location targeting set to "Presence: People in or regularly in your targeted locations."

Should You Bid on "TopLeft"?

Not yet. Competitor bidding is expensive and works best when you have a strong landing page with a direct comparison. TopLeft already has a comparison page against Proxuma. Bidding on their name without a counter-page risks driving traffic to TopLeft's comparison page via organic. Build a /proxuma-vs-topleft page first, then consider a small competitor campaign at EUR 2/day.

Should You Bid on "Autotask" Broadly?

No. "Autotask" alone is too broad (login queries, support queries, job seekers). Always pair Autotask with a feature term: "autotask project management," "autotask planning," "autotask outlook integration." The golden period proved this works. Broad "autotask" without feature qualifiers will burn budget on irrelevant clicks.

Critical Issue: Double/Triple Counting The account has 3 enabled conversion actions for the same booking event: a codeless page view ("Booked a demo"), another codeless page view ("bookingconfirm"), and a GA4 event ("bookingconfirmed"). The mu-plugin also fires a conversion event on the confirmation page. This means every single demo booking is counted 2-3 times, which inflates conversion numbers and confuses automated bidding. Similarly, recorded demo requests are double-counted (page view + GA4 event).

Recommended Fix

  1. Keep as PRIMARY (counting = "One"):
    • "Booked a demo (proxuma.io/)" - page view codeless (ID 7239093290)
    • "Page view (recorded-demo-requested)" - page view codeless (ID 7219446149)
  2. Set to SECONDARY (counting = "One", not used for bidding):
    • "bookingconfirmed" GA4 event (ID 7219607704)
    • "recordeddemorequested" GA4 event (ID 7219608715)
    • "Google Ads | Mail Click" (ID 7513798738)
  3. Remove or keep removed:
    • "bookingconfirm page view" (ID 7219443053) - duplicate, remove
    • All already-REMOVED actions - leave removed

After cleanup, you should have exactly 2 primary conversion actions: one for live demo bookings, one for recorded demo requests. This gives automated bidding clean signal.

mu-plugin Update Needed

The current mu-plugin (proxuma-google-ads-tracking.php) fires conversion events on 4 page IDs. Since we're using Google's codeless page-view tracking, the mu-plugin's conversion firing is redundant for pages 1937 and 1082 (which fire the same send_to as the codeless "Booked a demo" action). Consider removing the duplicate conversion calls from the mu-plugin to avoid triple-counting. Keep the mu-plugin for gtag.js loading only.

Missing: Portal Registration Tracking

Self-service signups at /portal/ (redirects to app.proxuma.io) are not tracked. If self-service trial starts are a conversion goal, add a conversion action for the app registration success page. This requires coordination with the app team to fire a gtag conversion event on successful registration.

6. 30-Day Launch Plan

Week 1 (May 5-11): Foundation
  • Day 1-2: Fix conversion tracking. Set duplicate actions to secondary. Verify only 2 primary actions remain. Test by booking a test demo and confirming exactly 1 conversion fires.
  • Day 1-2: Clean up campaigns. Pause all of John's campaigns permanently (Search | Leads | USA, Search | Leads | AUS). Remove or archive.
  • Day 3: Create new campaign structure. Build the 3 campaigns (Brand, Autotask Features, Recorded Demo) with the ad groups and keywords above.
  • Day 3: Add negative keywords at account level from the list above.
  • Day 4: Write and submit ads. Use the responsive search ad copy above. One RSA per ad group.
  • Day 5: Set budgets and bidding. Manual CPC, max CPC EUR 4 for Autotask Features, EUR 1.50 for Brand. Launch.
  • Day 5-7: Monitor. Check search terms report daily. Add negative keywords for irrelevant queries.
Week 2 (May 12-18): Optimize
  • Review search terms report. Add 10-20 negative keywords from actual queries.
  • Check which ad groups are getting impressions and clicks. Adjust bids up on performing ad groups, down on underperforming.
  • Verify conversion tracking is firing correctly (check Google Ads > Conversions > recent conversions).
  • Check Quality Score for each keyword. Score below 5 means the landing page or ad relevance needs work.
  • Set up Enhanced Conversions if not already active (sends hashed email data for better attribution).
Week 3 (May 19-25): Refine
  • By now you should have 100+ clicks. Analyze: which keywords got clicks? Which got conversions?
  • Pause any keyword with 50+ clicks and zero conversions.
  • Test a second RSA in the Project Management ad group (A/B test).
  • Check device performance. If mobile has zero conversions, apply -50% mobile bid adjustment.
  • Review time-of-day data. If all conversions happen during business hours, set ad schedule to Mon-Fri 7am-7pm.
Week 4 (May 26-Jun 1): Decide
  • 4-week performance review. Calculate actual CPA across all campaigns.
  • If CPA is under EUR 100 and you have 5+ conversions: switch Autotask Features to Maximize Conversions bidding with target CPA of EUR 80.
  • If CPA is over EUR 150 or zero conversions: review landing page quality, ad relevance, and keyword intent. Consider /planning/ page redesign before spending more.
  • If one ad group clearly outperforms others: reallocate budget from underperformers.
  • If "Outlook" or "Shifts" ad groups show promise: consider building dedicated landing pages (/outlook-integration/, /shifts/ improvement).
