Family 1: BROKEN_GRID (19 reports)

HTML references CSS classes that do not exist in the report's stylesheet. Common pattern: the report generator created custom classes like .doc-kpi-grid, .quadrant-grid, .garcia-card, .risk-grid, .heatmap-grid, .status-grid, or .cost-row in the HTML, but the corresponding CSS rules were never written. Content stacks vertically as unstyled text instead of forming grids, cards, or side-by-side layouts.

A sub-cluster of 4 NL project reports (budgetafwijking, gantt, winstgevendheid, taak-voltooiing) share a data bug: every hbar-fill shows the same value (50,751) so all bars hit 100% width. This is a content/data issue, not CSS -- the chart HTML itself contains wrong or placeholder data.

Why patches failed here Of the 19 BROKEN_GRID reports, 11 never received a patch because the original Opus vision scan returned patch_css: "". The problem was too structural for a CSS-only patch: when the HTML uses 15+ custom classes with no definitions, injecting CSS for all of them requires understanding the intended layout, which the vision model could not reliably infer from a screenshot alone. For the 8 that did get patches, the CSS addressed only part of the class mismatch -- patching .hbar-wrap but missing .hbar-item, for instance.

Family 2: EXPLODED_SVG (13 reports)

Inline SVG elements (chevrons, donut charts, code icons, checkmarks) render at viewport size because they lack explicit width/height attributes or CSS sizing. The SVG fills the entire teal frame or viewport as a giant black shape -- triangles (chevrons), circles (donuts), or brackets (code icons).

Why patches failed here All 13 received CSS patches targeting selectors like .dax-toggle-chevron svg or .faq-q svg. But many SVGs sit inside custom wrapper classes that vary per report -- .inline-dax .dax-trigger svg, .finding-icon svg, .pipeline-step-icon svg. The patches targeted the wrong selector, or the specificity was too low to override inline styles. The SVG sizing problem is simple (width: 16px; height: 16px;) but the selector targeting is report-specific.

Family 3: MISSING_KPI (6 reports)

Donut chart center text and KPI values are invisible. Root cause: SVG <text> elements use transform: rotate(90deg) but rotate around the SVG origin (0,0) instead of the element center, pushing the text off-canvas. This is a DOM attribute bug, not a CSS issue -- the transform-origin needs to be set per-element based on the text's position.

Family 4: OTHER (9 reports)

A mix of content bugs (wrong KPI labels showing company names instead of numbers), morph bar overlap issues, and reports where custom classes like .heatmap-cell or .cost-row are completely unstylled. One report (en__which-clients-are-actually-profitable) has a genuine content bug -- text labels where numeric KPIs should be.

Candidate Strategies

Recommended Strategy A: Template-Driven DOM Rebuild
Effort: 2-3 sessions | Coverage: 35-38 of 42

Instead of patching CSS onto broken HTML, regenerate the broken sections from golden-standard templates. The report generator created custom classes per report, but the underlying visual patterns repeat: KPI grids, hbar charts, donut charts, data tables, FAQ accordions. Build 6-8 section templates (kpi-grid, hbar-chart, donut-chart, data-table, faq-accordion, quadrant-grid, status-cards, finding-cards) with known-good CSS. Then for each broken report, identify which sections are broken, extract the data values from the existing HTML, and regenerate those sections using the template.

For EXPLODED_SVG: Replace all inline SVGs with a standard set of sized SVG sprites. One pass with a regex replacer handles all 13.

For MISSING_KPI: Fix the SVG text transform-origin with a targeted DOM edit (not CSS). 6 reports, same pattern.

For the 4 data-bug reports: Accept as unfixable by automation -- the underlying data is wrong (50,751 everywhere). Flag for manual review or exclude from gallery.

For the 1 content bug: (en__which-clients-are-actually-profitable) -- accept or manually fix. Label/value mismatch is not a rendering issue.

High coverage Reusable templates

Tradeoffs: Requires building and testing section templates, which is real work. But these templates also prevent future reports from having the same class-mismatch problem. The BROKEN_GRID family exists because the report generator creates novel class names per report -- templates eliminate that failure mode permanently.

Alternative Strategy B: Targeted Manual Patches
Effort: 1-2 sessions | Coverage: 30-35 of 42

Write hand-crafted CSS for each of the 42 reports individually. Open each report in a browser, identify the broken elements, write the CSS, verify. This is what Phase 8 already tried to do automatically -- doing it manually with human judgment on selectors and layout intent.

The EXPLODED_SVG family (13 reports) is fast: the fix is always the same pattern, just different selectors. The BROKEN_GRID family (19 reports) is slow: each report uses different custom classes and the intended layout must be guessed from surrounding context.

Manual labor No reuse

Tradeoffs: Fast to start, no infrastructure needed. But each fix is bespoke with no reuse. If the report generator creates new reports with the same class-mismatch pattern, they will break again the same way. Also, the 19 BROKEN_GRID reports require real layout judgment that is tedious to do 19 times.

Conservative Strategy C: Accept and Exclude
Effort: 30 min | Coverage: 0 (managed decline)

Mark all 42 reports as excluded from the gallery rotation. They remain on WordPress as individual pages but are not surfaced in the hero carousel or gallery grid. The 826 working reports (95.2% of corpus) are the public face. Revisit the 42 only if a client specifically requests one of those reports.

Optionally, add a "beta" or "preview" badge to these 42 pages so anyone who lands on them directly knows the rendering is imperfect.

Zero effort No fix

Tradeoffs: 95.2% is a strong coverage number. But the 42 include some prominent report topics (QBR, MRR, SLA Performance, Unprofitable Clients) that prospects might search for. Excluding them from gallery is fine, but they should still render correctly on their own pages.

Recommended Path

Go with Strategy A (Template-Driven DOM Rebuild) for the 35-38 fixable reports, combined with Strategy C (Accept/Exclude) for the 4-5 data/content bugs. The template approach fixes the current 42 and prevents the same failure mode from recurring in future reports. Start with the EXPLODED_SVG family (13 reports, one regex pass) since it is the fastest win, then MISSING_KPI (6 reports, targeted DOM edit), then BROKEN_GRID (19 reports, template rebuild). Leave the 4 data-bug reports and 1 content-bug report for manual review.
