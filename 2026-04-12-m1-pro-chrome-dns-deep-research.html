Deep Research — DNS Resolution Failure

M1 Pro Chrome is broken because Tailscale's daemon is a zombie

The WireGuard tunnel is alive (SSH works), but the Tailscale local daemon API has been dead for long enough that its DNS forwarder at 100.100.100.100 no longer answers. macOS still lists it as resolver #1, so every hostname lookup hits it first and times out.

Bottom line

The problem is not Chrome. It is not Surfshark. It is not Glide.

Tailscale's IPNExtension has been running for 12 days in a state where the WireGuard data plane still forwards packets (so the tunnel, and therefore SSH from the M4 Air, keeps working) but the control-plane localapi socket at 127.0.0.1:59695 refuses connections. That same daemon advertised itself to macOS as resolver #1 with order 111600 before it died. Every process that goes through getaddrinfo() (Chrome, curl, Python) blocks on the dead forwarder. Safari bypasses it using Network.framework's parallel resolver, which is why only Safari works. Fix Tailscale and everything else fixes itself.

The GUI process (pid 67084) and the extension (pid 67091) are both running, both started 2026-03-31, both burning ~140MB each. The tunnel interface utun4 is still UP with IP 100.85.64.20. But the localapi listener is gone, so nothing (not even tailscale set --accept-dns=false) can talk to it.

Resolver ordering confirms the trap

Lower order = higher priority. Resolver #1 is marked Supplemental with no search-domain restriction, so macOS sends every query to it first. The WiFi-configured Cloudflare DNS never appears in the active resolver list because Tailscale's accept-dns has replaced it.

Conclusion Network connectivity is perfect. TCP to port 443 on arbitrary public IPs completes in under 60ms. The ONLY thing broken is hostname-to-IP translation, and only for processes that use the standard system resolver chain. This fully explains why Chrome, curl, and Python all fail identically while Safari is fine.

Why the earlier fixes had no effect

Killing OrbStack freed RAM but did not touch DNS
OrbStack was a memory-pressure red herring. The DNS daemon had already died hours earlier. Memory is still tight (2.8GB swap used) but that is a separate problem.
Flushing mDNSResponder only invalidates the cache
dscacheutil -flushcache plus killall -HUP mDNSResponder drops cached entries. It does not change the resolver priority list, so the next query still goes to the dead Tailscale forwarder.
Passing the captive portal fixed Safari's trust, not the resolver
The Glide captive portal gate only had to be unlocked once for the network to allow egress. That is why Safari started working. Chrome's problem is upstream of egress.
Disabling Chrome's built-in DNS made things slightly worse
BuiltInDnsClientEnabled = false forces Chrome to use the C library resolver (getaddrinfo), which strictly honours resolver priority. Chrome's own resolver might have timed out faster and tried a fallback. Turning it off guarantees Chrome walks into the same trap as curl and Python.

Why Safari is the only survivor

Safari does not go through getaddrinfo. It uses Apple's Network.framework (nw_resolver_*), which runs parallel queries to every configured resolver and returns whichever answers first. Tailscale's dead forwarder never answers, so Safari simply uses Glide's answer a few milliseconds later without waiting. Chrome, curl, and python3 socket.getaddrinfo all funnel through mDNSResponder via the legacy resolver API, which serializes queries and waits for each timeout.

Fix options, ranked by safety and completeness

Safest Option 1 — Disable MagicDNS from the Tailscale admin console
Open login.tailscale.com/admin/dns on the M4 Air. Toggle off “Override local DNS” (and/or MagicDNS). The config propagates to the M1 Pro through the still-working data plane without needing the local daemon API to respond. SSH stays up because we touch nothing on the M1 Pro. After it takes effect, run dscacheutil -flushcache && sudo killall -HUP mDNSResponder over SSH. This is the cleanest fix short of a full restart.
Clean Option 2 — Restart the Tailscale app via a background trigger
SSH will drop for ~10-20 seconds while the WireGuard tunnel renegotiates. Use a detached command so the restart does not depend on the SSH session staying alive: nohup sh -c 'osascript -e "tell application \"Tailscale\" to quit"; sleep 6; open -a Tailscale' >/tmp/ts-restart.log 2>&1 &. After ~20 seconds the M1 Pro should be reachable again on 100.85.64.20 and the daemon will have a working localapi and a working forwarder.
Workaround Option 3 — Chrome DNS-over-HTTPS
In Chrome on the M1 Pro: chrome://settings/security → “Use secure DNS” → “With: Cloudflare (1.1.1.1)”. DoH resolves hostnames over HTTPS to an IP (1.1.1.1) that needs no DNS itself, so Chrome stops caring about the broken system resolver. Fixes Chrome only; curl and Python stay broken. Good as a same-minute unblock while you plan one of the real fixes.
Workaround Option 4 — Network Preferences DNS override
System Settings → Network → Wi-Fi → Details → DNS, set 1.1.1.1, 1.0.0.1, 130.43.187.40. This may not help on its own because Tailscale's supplemental resolver sits at order 111600 and still gets queried first. Worth trying because it takes 10 seconds, but do not count on it.
Risky Option 5 — Kill the IPNExtension process directly
kill 67091. The WireGuard tunnel collapses instantly, SSH disconnects mid-command, and you have to hope Tailscale auto-relaunches cleanly. If you are not physically at the M1 Pro (you are not, you are on the M4 Air), do not choose this. Only useful if Option 2 also fails.

Recommended sequence

  1. Flip “Override local DNS” off on the Tailscale admin console (Option 1).
  2. Wait 30 seconds, then over SSH: sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder.
  3. Test: curl -s -o /dev/null -w 'HTTP %{http_code} %{time_total}s\n' --connect-timeout 5 https://google.com. If it returns 301 in under a second, DNS is healed.
  4. If it still fails, fall back to Option 2 (background restart of Tailscale). Wait 20 seconds before retrying SSH.
  5. Once Chrome works again, consider re-enabling accept-dns later only if you actually rely on MagicDNS short names. You don't, as far as I can see from your workflow.
  6. Separately: schedule a weekly restart of Tailscale.app via launchd to prevent the daemon from zombifying again. 12 days is long enough that this is likely recurring.

Memory footnote

Swap is at 2.8GB of 4GB, 98K free pages (~1.5GB free). Better than the 660MB free starting point but still bad. Both Tailscale processes together hold ~250MB; Surfshark holds ~140MB in userspace plus a system extension. A Tailscale restart will transiently free its committed pages. After the DNS fix, consider quitting Surfshark too unless you actively need it, and consider a reboot if swap stays stuck above 2GB.

If it does, a weekly launchd restart is the right mitigation; if not, file a bug report with the output of log show --predicate 'subsystem == "io.tailscale.ipn.macsys"' for 2026-04-01 through today.
