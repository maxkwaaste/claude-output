M1 Pro Hardening Plan

Five fixes to ensure the M1 Pro server survives power outages, forced reboots, and kernel panics without manual intervention. Current uptime: 38 days. Goal: fully autonomous recovery.

Problem Summary

The M1 Pro has no recovery path after power loss or forced reboot. Three gaps compound: autorestart is not set (power outage = Mac stays off), auto-login is not configured (reboot = stuck at login screen), and Tailscale runs as a login item instead of a system daemon (no SSH until someone physically logs in).

Bottom line Any power blip today requires walking to the machine, pressing the power button, and manually logging in. All services (PostgreSQL, Ollama, Open Brain, PM2, Tailscale) are dead until then.

Risk Assessment

Scenario Current Outcome After Hardening
Power outage Mac stays off. All services down. No remote access. Physical intervention required. Auto-restarts, auto-logs in, services up in ~3 min. Monitoring alerts on failure.
Forced reboot Sits at login screen. Tailscale down, no SSH. LaunchAgents dead. Auto-login restores session. Tailscale daemon already running pre-login. Watchdog verifies in 2 min.
Kernel panic Reboots to login screen. Same as forced reboot. Same recovery path. macOS auto-restarts after panic by default.
Network blip Tailscale handles reconnection well. No issue. No change needed. Monitoring adds visibility.

The Five Fixes

Fix 1: Auto-restart After Power Loss
Zero Risk5 min

Single pmset command. Tells the SMC to power on the Mac automatically when AC power returns after a loss.

sudo pmset -a autorestart 1

Verify:

pmset -g | grep autorestart
# Expected: autorestart    1
Fix 2: Auto-login Configuration
Needs FileVault Decision10-30 min

Step 1: Check FileVault status.

fdesetup status
FileVault Decision If FileVault is ON, auto-login is impossible. FileVault requires pre-boot authentication. Recommendation: Disable it. This is a home server behind a router, not a laptop. The threat model does not justify disk encryption blocking automatic recovery. 
sudo fdesetup disable
# Decryption runs in background. Check progress:
fdesetup status

Step 2: Enable auto-login.

GUI (most reliable on macOS Ventura+): System Settings > Users & Groups > Automatic Login > select user.

CLI alternative:

sudo defaults write /Library/Preferences/com.apple.loginwindow autoLoginUser "max"
sudo sysadminctl -autologin set -userName "max" -password "-"

Verify:

defaults read /Library/Preferences/com.apple.loginwindow autoLoginUser
# Expected: "max"

Security trade-off: Home server behind NAT. Physical access = Max's home. SSH protected by Tailscale WireGuard auth. Auto-login risk is acceptable for this use case.

Fix 3: Tailscale as System Daemon
Medium Risk15 min

Migrate from Tailscale app (login item) to tailscaled (LaunchDaemon). The daemon starts at boot, before any user logs in.

Step 1: Record current state.

tailscale status
tailscale ip -4

Step 2: Install via Homebrew.

brew install tailscale

Step 3: Stop the app and remove from Login Items.

osascript -e 'quit app "Tailscale"'
# Then: System Settings > General > Login Items > remove Tailscale

Step 4: Start as daemon.

sudo brew services start tailscale

Step 5: Authenticate.

sudo tailscale up --operator=max
# --operator lets user 'max' control without sudo
# May print auth URL if re-authentication needed

Step 6: Verify.

tailscale status
tailscale ip -4
# Compare with Step 1 values
sudo launchctl list | grep tailscale
IP Change Risk The migration might assign a new Tailscale IP if the node identity changes (different state directory). If the IP changes from 100.85.64.20, update it in ~/.claude/projects/-Users-maxdekwaasteniet/CLAUDE.md on both machines and save to Open Brain.
Fix 4: Watchdog Service
Low Risk20 min

A bash script that checks all critical services and restarts any that are down. Runs as a LaunchDaemon (system-level, no login needed).

Pre-implementation discovery (run these first):

launchctl list | grep -E 'max|proxuma|rumble|brain|sdr|pm2'
which pm2
whoami

Services checked: PostgreSQL, Ollama, Tailscale, PM2, Open Brain (port 9876), Rumble Server, SDR Dashboard (port 3849).

Two LaunchDaemons:

  • com.proxuma.server-watchdog — runs every 30 min, checks all services, restarts failures
  • com.proxuma.server-watchdog-boot — runs once at boot after 2-min delay, catches post-reboot issues

Full script and plists in the plan file at ~/ClaudeCode/m1pro-resilience-plan.md.

Fix 5: Remote Monitoring
Low Risk15 min

Two layers of monitoring:

  • Layer 1 (M1 Pro): Heartbeat to healthchecks.io every 5 min. Free tier, 20 checks. If heartbeat stops, sends email/push alert.
  • Layer 2 (M4 Air): Independent Tailscale ping check every 5 min. After 2 consecutive failures (10+ min), shows macOS notification with sound.
