Board Setup & Architecture

How Proxuma's generic KanBan system works. Two live adapters (Deal Pipeline, Development Backlog), fully configurable columns and swimlanes, and the adapter pattern that makes it extensible to any entity type.

Architecture: Adapter Pattern

One generic VKanBan component set, multiple entity-specific adapters. The board doesn't know what it's displaying -- the adapter tells it.

// Shared components (entity-agnostic) components/shared/VKanBan/ VKanBanBoard.vue -- Main container (swimlanes, flat, drag context) VKanBanColumn.vue -- Column with DnD + pagination (10 cards) VKanBanCard.vue -- Configurable card with expandable fields VKanBanFilters.vue -- Filter panel VKanBanViewSelect.vue -- Board selector (CRUD, duplicate, default) VKanBanViewSettings.vue -- Side panel: columns, swimlanes, card layout VKanBanDropOverlay.vue -- Viewport-fixed drop overlay VKanBanSwimlanesToggle.vue -- Quick swimlane switching // Stores (Pinia, entity-agnostic) stores/kanban/ kanbanBoardStore.ts -- Board config CRUD kanbanDataStore.ts -- Data fetching via adapter kanbanFilterStore.ts -- Filter state (localStorage persisted) // Adapters (entity-specific) types/kanban/ dealAdapter.ts -- CRM deal pipeline jiraAdapter.ts -- Development backlog (read-only)
Adapter interface
Each adapter defines: columnOptions, swimlaneOptions, cardFieldDefinitions, cardPresets, defaultColumns, canCrossSwimlane(), getDropTargets(), getCardLink(). Adding a new entity type means writing one adapter file.

Backend: Laravel 11+ / PHP 8.4 / PostgreSQL 16. Board configs stored in forge.saved_views as JSONB, per user, per module. Deep merge updates so partial config changes don't overwrite the full board.

Read-only adapter

Board Features

FeatureImplementation
WIP LimitsPer-column limit. Visual indicator (red count) when exceeded. Configured in BoardSettingsColumns.
Auto-refresh30-second polling. Silent background update.
Pagination10 cards per column. "Show more" to load next page.
Hide completedConfigurable hideCompletedAfterDays (0-365). Done/Won/Lost cards auto-hide.
Saved viewsPer user, per module. CRUD + duplicate + set default. Stored in forge.saved_views JSONB.
Card presetsAdapter-defined field sets. Deal: "Pipeline Manager" (4 fields), "Minimal" (2 fields).
Settings panelBoardSettingsColumns (add/remove/reorder/WIP), BoardSettingsSwimlanes (5+ options), BoardSettingsCardLayout (field visibility + zones).
Optimistic updatesDeal adapter only. Instant UI update + rollback on API failure.
DnD approachNative HTML5 draggable="true" + custom ghost. Composables: useKanbanDnd, useKanbanCardDrag, useAutoScroll.

Future Adapters

The adapter pattern makes adding new entity types straightforward. One TypeScript file per adapter.

Planned AdapterEntity TypeSourceNotes
TaskstaskAutotask / internalProject tasks on a kanban board
ProjectsprojectAutotask / internalProject phases as columns
CompliancecomplianceInternalAudit items through review stages
Architecture note
The VKanBan components are fully entity-agnostic. Adding a new adapter requires: one adapter file defining columnOptions, swimlaneOptions, cardFieldDefinitions, and a fetchCards/moveCard implementation. No changes to shared components.
