Option A: Pure Excel Formulas
FEASIBILITY: LOW
The core problem is that this requires ETL logic, not spreadsheet math. You need to: (1) scan variable-length ranges for non-empty rows, (2) detect phase boundaries by matching text patterns in column B, (3) track which phase each task row belongs to, and (4) produce dynamic-length output. Excel 365 has FILTER/SEQUENCE/LET that can handle some of this, but phase tracking across sparse data with formula-based phase headers (=IF(Global_Taal="NL",...)) makes it extremely fragile. Any structural change to the Project Services sheet breaks the formulas. Estimated: 40+ complex formulas, undebuggable by Cheops staff.
HIGH COMPLEXITY FRAGILE NOT RECOMMENDED
Option B: VBA Macro Button
FEASIBILITY: HIGH
A VBA macro on a new "AT Import" sheet reads all 7 tables, detects phases, builds the 20-column output, and writes it to the sheet. User clicks a button, gets the AT import data, copies to CSV. Clean, debuggable, maintainable. The catch: the calculator is .xltx (no macros). It would need to become .xltm (macro-enabled template). This changes the file type but not the workflow. Estimated: 120-150 lines of VBA.
RECOMMENDED CLEAN REQUIRES .XLTM
Option C: External Python Script
FEASIBILITY: HIGH
A standalone Python script (using openpyxl) reads the filled-in calculator and produces a CSV file. Easiest to build and most reliable. But it adds a dependency: Cheops needs Python installed, or someone runs the script for them. This breaks the "single file" promise. Better suited as a Dxfferent internal tool than as something delivered to Cheops.
INTERNAL TOOL RELIABLE EXTERNAL DEPENDENCY

Tricky Parts

  1. Formula-based phase headers. The "Documentatie & Project" text in column B is an =IF(Global_Taal="NL",...) formula, not plain text. Any solution must resolve these formulas or match on the formula pattern. VBA handles this natively (reads calculated values). Formulas and Python (with data_only) need workarounds.
  2. Empty template rows. Each project table has ~25 rows pre-allocated. When presales uses only 8 tasks, the other 17 rows are empty placeholders (just "Cheops" in column D and a role-rate in G). The solution must skip these cleanly.
  3. Hours 150% column. Some cells contain a space character (' ') instead of being truly empty. Must handle this as zero.
  4. Variable phase count. Project 1 has 3 phases + Doc. Projects 2-6 have 4 phases + Doc. The Managed Services table has 6 completely different sections. Can't hardcode phase counts.
  5. Date placeholders. The calculator has no date information. All dates in the AT export must be placeholders that the PM fills in manually. This is a workflow step that can't be automated away.
  6. Project naming. The calculator labels them "Project 1" through "Project 6". In practice, presales fills in a project name elsewhere (the BOM sheet). The AT export should ideally use that name, not "Project 1".

Recommendation

Go with Option B (VBA macro), but deliver it alongside the existing separate template approach. The VBA solution is the cleanest way to embed AT export directly into the calculator. The .xltx to .xltm change is minor and transparent to users. The macro reads live calculated values, handles formula cells natively, and produces clean output. Build time: half a day. Deliver the Python prototype (Option C) as a Dxfferent internal fallback for when the macro approach hits edge cases.

Prototype Status

A working prototype has been generated at:

~/ClaudeCode/dxfferent/cheops-autotask-import/source-files/Calculator + AT Export Prototype.xlsx

This prototype was built with Python/openpyxl and demonstrates the full mapping logic. It adds an "AT Import" tab to the calculator with:

  • All 20 Autotask import columns with correct headers
  • 34 tasks extracted from the template (only pre-filled rows, since the template is empty)
  • Proper phase detection including formula-based "Documentatie & Project" headers
  • Empty placeholder row filtering
  • Role-rate to Work Type mapping
  • Status logic (Te plannen vs New based on hours)
  • Source row references in a notes column for traceability
Next step To turn this into the VBA solution, the logic from the Python prototype needs to be rewritten as a VBA Sub attached to a button on the AT Import sheet. The template file type changes from .xltx to .xltm. The mapping logic stays identical. The VBA version will also pick up the actual project names from the BOM sheet and read calculated formula values directly.
