What Each File Is

1. Calculator & Presales Template (REV9.34)
CHEOPS SALES TOOL - 22 SHEETS
Cheops's internal presales calculator. Sales and presales fill in customer details, cloud requirements, managed services parameters, and security options. The template computes a BOM, pricing overview, and contract tables. The "Project Services" sheet defines hourly rates (Support Engineer: 115/h, System Engineer: 140/h, Senior/PM/Architect: 155/h) and project hours per role. This is where project scope gets born, but it has zero Autotask structure -- no phases, no task fields, no import format.
Sales origin No AT structure
2. PIA - T&M - Migratie BFL - 202601.csv
CHEOPS AUTOTASK EXPORT - SAMPLE PROJECT
A CSV export from Cheops's own Autotask instance showing a real M365 migration project ("PIA - T&M - Migratie BFL"). Contains 3 phases, 13 tasks, columns for Alert, Issue Icon, Milestone Icon, Phase/Task/Issue, Task Category, Estimated Hours, Duration, Start Date, End Date, Work Type, Resource(s). This is Cheops's current Autotask import format. The structure is flat: phases as top-level rows, tasks indented. Hours use comma decimals (e.g., "16,00"). All dates are DD/MM/YYYY.
AT import format Real data
3. Dxfferent Master Template (May 2026)
BEST PRACTICE - 47 TASKS, 8 PHASES
Our master template for what a well-structured Autotask project looks like. 8 phases covering the full project lifecycle from preparation through finalization. 20 columns mapped to Autotask import fields: Phase Name, Phase Start/End, Task Title, Task Start/End/Time, Estimated Hours, Department, Work Type, Task Category, Primary Resource, Priority, Status, PO Number, Client Portal Access, Issue, Task Type, Start No Earlier Than. Column V has implementation notes (not imported). Every task has concrete scope and hour estimates.

Phase Structure Comparison

Master Template (8 phases)

  1. Project preparation (8 tasks) -- Assessment, project plan, hardware orders, impact analysis, kick-offs
  2. Project Administration (3 tasks) -- PM, travel time, meetings (hours = 0, time tracking only)
  3. Additional work (1 task) -- Buffer for scope changes
  4. Design and PoC (3 tasks) -- Design, test, acceptance
  5. Implementation - Core/Cloud/Workstation/Security (4 sub-phases, 18 tasks) -- Configure, implement, accept, migrate per domain
  6. Location infrastructure (4 tasks) -- Network, onsite, acceptance, migration
  7. Final delivery and adoption (4 tasks) -- Testing, control, IT Glue docs, training
  8. Finalization and Support (5 tasks) -- Aftercare, transfer to support, contracts, evaluations

Cheops Sample (3 phases)

  1. M365 migratie (3 tasks) -- Voorbereiding, presync, cutover
  2. Onsite Aftercare (1 task) -- OPTIONEEL
  3. Project Management & Documentation (6 tasks) -- Kick-off, onboarding, handover, meetings, docs, PM

Cheops uses a minimal phase structure focused on the specific delivery, while the master template covers the full lifecycle from assessment to aftercare. Both approaches are valid, but the master template gives much better visibility for resource planning and progress tracking.

What We Can Do for Cheops

Option A: Template Mapping + Import Guide

Low effort, immediate value
EFFORT: 2-4 HOURS
Create a Cheops-specific version of the master template pre-filled with their common project types (M365 migration, cloud hosting, managed services onboarding). Map their Calculator template's "Project Services" sheet outputs to the right Autotask import columns. Provide default values for the fields they don't currently fill in. Deliver as an Excel template they can copy-paste from, plus a step-by-step import guide.

Option B: Autotask Project Templates (Native)

Medium effort, structural improvement
EFFORT: 4-8 HOURS
Set up project templates directly inside Autotask. Create 3-5 templates covering their main project types (M365 Migration, Cloud Hosting Setup, Managed Services Onboarding, Security Implementation, Full Infrastructure). Each template pre-configured with phases, tasks, default hours, work types, and departments. PMs then create projects from templates instead of spreadsheets, and only need to adjust dates and resources. Combined with Proxuma for visual planning after creation.

Option C: Sales-to-Project Automation

High effort, maximum value
EFFORT: 16-24 HOURS
Build a workflow where the Calculator template's output (project type, hours per role, scope items) automatically generates an Autotask project via the API. The presales template already captures project type, hour estimates, and scope. Map those fields to the right Autotask project template. Could use Power Automate, a simple web form, or a custom script that reads the Calculator output and calls the Autotask REST API. This eliminates the manual translation step entirely.
Recommendation Start with Option B. Autotask native templates are the right mid-ground. They give Cheops immediate structure without requiring them to change their sales process. Option A is a band-aid (still spreadsheet-based). Option C is the end goal but requires buy-in on the Calculator template format staying stable. Option B gets them operational in Autotask within a week, and Proxuma handles the visual planning layer on top.

PS Planning-Prognose: What It Tells Us

This file is Cheops's resource planning spreadsheet. 20 system engineers, daily planning from Jan 2026 onward, 1241 rows (roughly 6 months of workdays). Each engineer gets two columns (AM/PM slots). Entries are project names + client names ("Studibo - data migratie Ingeo", "Digitopia - Migratie NFGD", "Prof Praxis - Redundante Server infra"). Also tracks leave ("Verlof"), sick days ("Ziek"), and holidays ("Nieuwjaar").

This is exactly what Proxuma replaces. Once projects live in Autotask with proper tasks and resources, Proxuma's capacity view gives them this same information without maintaining a separate Excel file. The planning-prognose spreadsheet is the strongest argument for why they need to move to Autotask-native project management.

Bottom Line

Cheops has all the raw data in their Calculator template but no bridge to Autotask. The gap is structural, not technical: they need project templates with the right phases and tasks, not better spreadsheets. The master template gives us the blueprint. The Autotask import format is well-defined (they already use it). The real win is getting them off the planning-prognose spreadsheet and onto Proxuma's capacity view, which requires projects to actually exist in Autotask with proper task structures.

  • 5 missing import fields all have safe defaults -- no blockers
  • Start with native AT templates (Option B), graduate to API automation later
  • The planning spreadsheet is Proxuma's strongest sales argument here
