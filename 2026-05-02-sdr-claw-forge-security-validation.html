SDR Claw Forge Security: Spec vs. Reality

Three Opus 4.7 agents validated the Forge security proposals (token scoping, anonymization, email pipeline) against the actual nexum-backend codebase. This report captures what exists, what is missing, and what needs to change in the spec.

1. Token Scoping: Infrastructure Exists, Just Disabled

Key finding The research spec proposed building token scope middleware from scratch. In reality, the nexum-backend already has HasScopedApiTokens and TokenScopeIntersect middleware built and wired up. They run in dry-run mode (log only, no deny). Flipping one env var activates enforcement.

Revised Estimate

The spec estimated 30 minutes for Joshua. With the existing infrastructure, the actual work is closer to 15 minutes:

  1. Set PLATFORM_TOKEN_SCOPE_INTERSECT_ENABLED=true in .env
  2. Add allowed_ips migration + 10 lines in TokenScopeIntersect
  3. Add per-token rate limiter in RouteServiceProvider
  4. Issue the SDR Claw token with exact abilities (the createToken() call from the spec works as-is)
Warning Enabling PLATFORM_TOKEN_SCOPE_INTERSECT_ENABLED affects ALL tokens, not just the Claw's. Every existing user token currently has ['*'] abilities, so the intersection logic will work fine. But if any token was issued with limited abilities, they would suddenly be enforced. Test in staging first.

2. Forge-Level Anonymization: Clean Slate, Good Fit

Key finding No anonymization exists in the codebase. The only PII handling is LoggableContextProcessor which redacts 28 sensitive keys in log output. The Forge controllers return raw Eloquent models with no API Resource transformation layer.

Infrastructure Ready

DependencyStatusEvidence
RedisReadySession, cache, queue all on Redis. Horizon running 4 supervisors. docker-compose has redis:alpine with health check.
Service layerReady19 modules in app/Services/. Constructor injection with readonly promoted properties. app/Services/Forge/ exists with 1 service.
Cache facadeReadyCache::remember() used throughout for Graph tokens, rate limiting. Redis-backed.

Architecture Decision: Where to Hook Anonymization

The spec proposed a ?view=safe query parameter on existing endpoints. Three options for implementation:

Option A: Middleware
Recommended
Intercept JSON responses for tokens with read:safe ability. Transform PII fields before the response leaves the controller. No controller changes needed. Works with raw Eloquent returns.
Low invasionToken-aware
Option B: API Resources
Cleaner but bigger scope
Add Forge API Resource classes for all 16 controllers. Resources check token scope and conditionally redact. Better long-term, but touches every controller return statement.
High invasionBetter DX

Recommendation: Option A (middleware). The Forge controllers already return raw Eloquent without API Resources. Adding a response-transforming middleware is non-invasive and can be scoped to tokens with read:safe ability only. Regular tokens see no change.

Presidio Sidecar: Hybrid Approach Confirmed

The spec proposed a hybrid: PHP for structured fields, Python Presidio for free-text (transcripts, notes). This is the right call. The codebase already has Python in the stack (PostHog sync), and the powerbi-claude-skills repo has a working Presidio implementation to reference.

4. Spec Gaps: What the Research Got Wrong

Gap 1: PostHog Tenant Matching Does Not Exist
The spec assumes company.domain -> posthog tenant_slug matching. In reality, there is no mapping table or mechanism between PostHog tenant slugs and CustomerRegistry records. The tenant_slug lives only in analytics views. A new join/mapping needs to be built, or the Claw must do fuzzy domain matching locally.
Spec incorrectNeeds new code
Gap 2: Enrichment Fields Not in Template Variables
The spec references pain_points, buying_signals, decision_authority in email context. These fields exist in forge.contact_enrichments and forge.company_enrichments, but they are not available as template variables. The SDR Claw cannot use them in email personalization without extending renderTemplate().
Needs extension
Gap 3: Token Scope Middleware Already Built
The spec proposed building EnforceTokenAbilities middleware from scratch. The codebase already has TokenScopeIntersect with wildcard support and Spatie permission integration. The spec's PHP code should be replaced with config changes to the existing middleware.
Spec overcomplicated
Gap 4: Campaign Send is Async (Queued)
The spec implies synchronous send. In reality, sendCampaign() dispatches SendCampaignJob to a Redis queue (10-min timeout, 3 retries, 60s/300s backoff). The Claw gets a 202 back, not confirmation that all emails sent. The Claw needs to poll campaign status or subscribe to a webhook for completion.
Spec incorrect
Gap 5: No Forge API Resources
The anonymization proposal assumed API Resource classes exist for transformation. All 16 Forge controllers return raw Eloquent models. Anonymization must use response middleware instead of resource-level transforms.
Architecture mismatch

6. Revised Implementation Plan for Joshua

Phase 1: Token Scoping (15 minutes)

  1. Migration: add allowed_ips JSON column to personal_access_tokens
  2. Add IP check to existing TokenScopeIntersect middleware (10 lines)
  3. Add per-token rate limiter in RouteServiceProvider (15 lines)
  4. Set PLATFORM_TOKEN_SCOPE_INTERSECT_ENABLED=true in staging, test
  5. Issue Claw token: $user->createToken('sdr-claw', [...exact abilities...], now()->addDays(90))

Phase 2: Anonymization Middleware (2 days)

  1. Day 1: ForgeAnonymizationMiddleware that intercepts JSON responses for tokens with read:safe ability. Field-level rules for structured data (financials, email, phone). Redis session map with 24h TTL.
  2. Day 2: Presidio sidecar for free-text fields (transcripts, notes). Deanonymization in GraphMailSendService::sendSingle() before template rendering. Testing, edge cases.

Phase 3: Pipeline Extensions (1 day)

  1. PostHog tenant-to-company mapping table or view
  2. Extend renderTemplate() with enrichment variables
  3. Add campaign completion webhook/status polling endpoint for async send tracking

Bottom Line

The original spec overestimated the middleware work (it already exists) and underestimated the anonymization complexity (no API Resources to hook into). Token scoping is a config change + one migration. Anonymization is a response middleware pattern, not a controller-level transform. The biggest missing piece is PostHog tenant matching, which is not a security concern but blocks the upsell email category entirely. Total revised estimate: 3 days for Joshua instead of 3.5 days.
