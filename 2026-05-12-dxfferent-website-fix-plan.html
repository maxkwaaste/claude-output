ACF step-text block • missing block, not just a text fix
Both EN and NL "How We Work" pages skip from Step 3 (Kick-off) to Step 5 (Go-live). The MSP Journey page does have a Step 4 (Gameplan: goals and strategy), but How We Work is a different process description (implementation-focused, not assessment-focused).

9. Je/uw consistency across NL pages
10+ pages • editorial style decision
Multiple NL pages mix informal "je/jij/jouw" with formal "u/uw" in the same paragraph. Example: "Van het optimaliseren van je Autotask tot het opnieuw uitvinden van uw dienstenaanbod." This appears on at least pages 702, 1529, 1971, 1976, and 766.

The site's overall tone is informal (matching the MSP market in NL). Recommendation: standardize on "je/jij/jouw" everywhere and remove all "u/uw" instances.

Scope: This is a significant content sweep across 10+ pages. Each replacement needs context-checking (some "uw" might be in testimonials or legal text where formal language is appropriate).

Decision: Flag for a separate session. Not a quick fix.
Deferred

Execution approach

Key constraint The entire site uses ACF blocks with data stored as JSON in post_content. Every edit is a str_replace or regex on that JSON via $wpdb->update(). One wrong character in the JSON breaks the page. Every fix gets a backup first, and a rendered HTML check after.

Execution order

  1. Items 1-2 (IT Glue garbled text + MSP Journey truncated title): Simple str_replace on known strings. No structural risk.
  2. Item 3 (Security Officer duplicate): Replace duplicate text with new sentence. Changing a value in existing JSON, not adding/removing structure.
  3. Item 4 (About page content): Update the hero text field. Page structure stays the same.
  4. Items 5-6 (Footer links): Most impactful. Back up both global blocks first. Add one link each to tab_1_links_row1 arrays. Verify all pages render correctly.
  5. Item 7 (Proxuma description): Investigate products block template first, then fix.

Verification per fix

  1. Back up page content: wp post get ID --field=content > /tmp/backup_ID.html
  2. Apply fix via $wpdb->update()
  3. Flush cache: wp cache flush && wp sg purge
  4. Verify via REST API: fetch page content and check fix is present
  5. Verify via curl: fetch rendered HTML and confirm page is not broken
  6. Open in browser for visual check

Bottom line

7 items can be done autonomously right now. Items 1-4 are safe text replacements. Items 5-6 (footer) are high-impact but mechanical. Item 7 (Proxuma) needs one more investigation step. Items 8-9 need editorial decisions before execution.
