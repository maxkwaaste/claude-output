Context SDR Claw is an AI-powered email outreach pipeline built into NeXum Forge. Two Claude scheduled agents (morning drafts + evening learning) talk to Forge via REST API. Forge owns the review UI, email sending, storage, and sequence timers. No external dependencies beyond the existing Claude Max subscription and MS Graph connector. Deploy to M1 Pro
Estimated: 1-2 hours

Deploy backend and frontend to the live Forge server on M1 Pro. The server runs at /var/www/nexum/ (accessible via ssh hetzner or direct Tailscale SSH to M1 Pro depending on your deploy setup).

  • Push feature branches to forks -- push feature/sdr-module on maxkwaaste/nexum-backend and feature/sdr-review-page on maxkwaaste/nexum-frontend to GitHub. Needed to pull from server. Max
  • SSH into server and pull backend branch
    cd /var/www/nexum/backend && git remote add maxfork https://github.com/maxkwaaste/nexum-backend.git && git fetch maxfork && git checkout -b feature/sdr-module maxfork/feature/sdr-module
    Or if deploying from local clone: merge feature branch into whatever branch the server tracks. Max
  • Run the migration
    php artisan migrate
    Creates sdr schema + 5 tables (batches, drafts, feedback, writing_rules, sequences). Verify: php artisan tinker --execute="echo count(DB::select(\"SELECT tablename FROM pg_tables WHERE schemaname='sdr'\"));" should return 5. Claude
  • Pull frontend branch -- same pattern as backend. Rebuild: cd /var/www/nexum/frontend && npm run build Max
  • Create SDR Sanctum token
    php artisan tinker then: $user = User::where('email', 'max@proxuma.com')->first(); echo $user->createToken('sdr-agent', ['sdr.read', 'sdr.write'])->plainTextToken;
    Store the token. It goes in the Claude scheduled agent config and in .env as SDR_AGENT_TOKEN. Max
  • Add Telegram config to .env
    TELEGRAM_BOT_TOKEN=your-bot-token
    TELEGRAM_CHAT_ID=your-chat-id
    Add to config/services.php: 'telegram' => ['bot_token' => env('TELEGRAM_BOT_TOKEN'), 'chat_id' => env('TELEGRAM_CHAT_ID')] Claude
  • Register artisan schedules -- verify sdr:check-sequences and sdr:alert-missing-batch appear in php artisan schedule:list. Confirm Laravel scheduler cron is running on the server. Claude
  • Clear caches
    php artisan config:clear && php artisan route:clear && php artisan cache:clear Claude
2
Validate End-to-End
Estimated: 1-2 hours

Test every endpoint and UI flow against real Forge CRM data. Nothing gets sent to real contacts until Phase 3.

API Smoke Tests (curl with SDR token)

  • GET /api/sdr/candidates -- should return up to 50 pre-ranked contacts from the CRM. Check that it excludes unsubscribed contacts and contacts already emailed today. Claude
  • GET /api/sdr/candidates/{id}/deep-context -- pick one contact ID from the candidates response. Should return contact details, deals, recent emails, recent meetings. Claude
  • GET /api/sdr/learnings -- should return empty rules (fresh install) and empty approval stats. Claude
  • POST /api/sdr/batches -- push a test batch with 2-3 drafts using real contact IDs from the candidates response. Use category stale_deal_nudge. Verify Telegram notification arrives. Claude
  • GET /api/sdr/batches/{today's date} -- should return the batch with drafts. Claude

Review UI Tests (browser)

  • Navigate to forge.proxuma.io/sdr/review -- should load the review page with the test batch. Check: category KPI cards show, draft cards render with subject/body/score. Max
  • Test keyboard shortcuts -- J/K to navigate, A to approve (send), S to skip, Z to snooze. Max
  • Test edit flow -- modify subject and body text in one card, then press A. Verify the sent email (check Outlook sent folder) has the edited text, not the original. Max
  • Verify email actually sends -- approve one draft for a test contact (test@proxuma.com or Max's own email). Check Outlook sent folder. If it doesn't send, check GraphMailSendService config and MS Graph token. Max
  • Verify feedback recording -- after approving/editing/rejecting a few drafts, check GET /api/sdr/feedback-stats. Edit diffs should show non-zero edit_distance for edited drafts. Claude

Candidate Query Validation

  • Check the SQL -- the candidate service queries canonical.contacts, canonical.deals, canonical.mail_signals, canonical.calendar_events, forge.contact_enrichments. Verify these tables exist and have data on the production DB. If table names differ, fix SdrCandidateService.php. Claude
  • Verify payment data exclusion -- run a test query and confirm no Stripe IDs, IBAN, or invoice amounts appear in the response. Claude

Daily Use (3-5 Business Days)
Estimated: Mon May 5 through Thu May 8

Run the full loop every workday. Start manually (Claude posts a batch at 07:00), review in the UI, send/skip/snooze. The learning agent runs at 17:00 to pick up edit patterns.

Day 1 (Mon May 5) -- Manual Batch
  • Run sdr-morning agent manually (not scheduled yet). Have Claude fetch candidates from the API, draft 10-15 emails (not 25, start small), and POST the batch.
  • Review all drafts in the UI. Edit freely. Pay attention to what you change: this feeds the learning loop.
  • Send 5-8 emails to real contacts (low-risk: stale deals, win-backs older than 90 days).
  • Run sdr-learning agent manually at end of day. Check that writing rules appear.
Day 2 (Tue May 6) -- Expand + First Learnings
  • Run morning agent again. It now has Day 1's writing rules. Check if drafts improved.
  • Expand to 20 drafts. Include post-demo and inbound categories.
  • Check Outlook for any replies to Day 1 emails. Note response rate.
  • Run learning agent. Rules should grow from Day 1+2 combined feedback.
Day 3-4 (Wed-Thu May 7-8) -- Full Volume
  • Full 25-draft batches. All 5 categories active.
  • Track: approval rate, edit rate, rejection rate per category.
  • Configure the scheduled agents for real (cron via claude schedule create) if manual runs are working reliably.
  • Test the sequence flow: Day 3 follow-ups should appear for contacts approved on Day 1.
  • Verify sdr:check-sequences artisan command moves pending to queued at 06:00.
Day 5 (Fri May 9) -- Assessment
  • Review the week's stats: total drafts, approval rate, edit ratio, emails sent, replies received.
  • Check writing rules: do they make sense? Are any wrong?
  • Fix any bugs found during the week.
  • Decision: ready for PR or need another week?
Success Criteria for Phase 3
  • Approval rate above 50% by Day 3 (approved + edited, not rejected/snoozed)
  • At least 5 writing rules with confidence above 0.6
  • Email send via MS Graph works reliably (no send failures)
  • Sequence follow-ups appear correctly on Day 3
  • No data leaks (payment/accounting data never surfaces in SDR endpoints)
  • Review UI works smoothly with keyboard shortcuts

Risk Mitigation

Wrong emails go out
Every email requires manual approval in the review UI. Nothing sends automatically. The agent drafts, Max reviews. Worst case: Max approves something he shouldn't have. Mitigation: start with low-risk categories (stale deals, win-backs) before touching post-demo and inbound.
SQL query hits wrong tables
The candidate service uses raw SQL against canonical.* and forge.* schemas. If table names differ on the production DB, the endpoint returns errors. Mitigation: Phase 2 smoke test catches this immediately. Fix is a 5-minute SQL edit.
Migration conflicts with other work
The SDR migration creates its own schema (sdr.*), completely isolated from existing tables. Zero risk of conflicting with ongoing WBSO, marketing, or CRM migrations.
Joshua rejects the PR
The 3-5 day test period produces real usage data. The PR includes actual approval rates, edit patterns, and response metrics. This is stronger evidence than code review alone. If Joshua wants architectural changes, the sdr schema isolation makes refactoring contained.

Bottom Line

The code is written. The risk is low because everything requires manual approval before sending, the SDR schema is fully isolated, and the NeXum deploy-first workflow means you validate on real data before committing. Start Monday May 5, test through Thursday May 8, open PR on Friday May 9.
