Hosting Decision: Hetzner vs M1 Pro

Key clarification NeXum/Forge production runs on Hetzner (at /var/www/nexum/), not M1 Pro. The Hetzner server runs PHP-FPM (backend) + PM2 (frontend) and serves forge.proxuma.io, venture.proxuma.io, and my.dxfferent.nl. M1 Pro is Max's home server for automation services (PostgreSQL, Ollama, Open Brain, LaunchAgents), not for hosting NeXum.

Two options for go-live:

Option A: Deploy to Hetzner (recommended)
Standard NeXum workflow
Pull feature branches on the Hetzner server. Migration runs against the production PostgreSQL. Email sending works immediately (Azure AD credentials + sender mailbox already exist). All CRM data is already there. No extra infrastructure. Joshua's standard deploy-first, commit-after workflow.
Option B: Full Copy on M1 Pro
Isolated test environment
Clone the entire NeXum stack locally. Requires: PostgreSQL 16 with full DB dump, Redis, PHP-FPM, PM2, Azure AD credentials replicated or new app registration, sender mailbox seeded. Safer for testing (no production risk) but more setup work and RAM-constrained.

RAM verdict M1 Pro with 16 GB can run a full local NeXum copy, but only if Ollama models are not loaded simultaneously.   • Clone Azure AD app credentials into canonical.platform_credentials (platform='graph', account='proxuma'). Or register a separate Azure AD app for local testing.
  • Seed forge.email_sender_mailboxes with max@proxuma.com (is_active=true).
  • PHP 8.4 + extensions: brew install php@8.4 php@8.4-pgsql php@8.4-redis. Install Composer. Run composer install.

  • Configure nginx or use php artisan serve for quick testing (not production-grade).

Outlook Integration: Prerequisites

SDR Claw sends emails via GraphMailSendService, which uses Microsoft Graph API with app-only (client_credentials) auth. This is the same service the existing Forge email campaigns use. It sends from Max's mailbox and saves to Sent Items.

PrerequisiteWhat It IsStatus
Azure AD App RegistrationApp with Mail.Send permission (application type, not delegated). Grants ability to send as any user in the Dxfferent/Proxuma tenant.Already exists (used by Forge email campaigns on Hetzner). Credentials encrypted in canonical.platform_credentials.
PlatformCredential rowRow in canonical.platform_credentials where platform='graph', account='proxuma'. Contains encrypted tenant_id, client_id, client_secret.Exists on Hetzner production DB. If running locally on M1 Pro, needs to be imported or recreated.

How email sending works
  1. Max approves a draft in the review UI (clicks Send or presses A)
  2. Frontend calls POST /api/sdr/drafts/{id}/approve with optional edited subject/body
  3. SdrReviewController calls SdrFeedbackService::approveDraft()
  4. Service records feedback, then calls GraphMailSendService::sendSingle('max@proxuma.com', contact_email, subject, body)
  5. GraphMailSendService loads Azure AD token from PlatformCredential, validates sender mailbox, calls Graph API /users/max@proxuma.com/sendMail
  6. Email appears in Max's Outlook Sent folder. Graph API response confirms success.
  7. Service creates signal_entity_links in canonical schema (contact/deal association for the sent email)

No MCP, no delegated auth, no browser login. Fully server-side. The same pipe that Forge email campaigns already use.

Option A: Deploy to Hetzner (recommended)

Deploy directly to the production Hetzner server at /var/www/nexum/. The SDR schema is fully isolated, email sending works out of the box, and all CRM data is already there.

  • Push feature branches to forks -- push feature/sdr-module on maxkwaaste/nexum-backend and feature/sdr-review-page on maxkwaaste/nexum-frontend to GitHub. Max
  • SSH into Hetzner and pull backend branch
    ssh hetzner
    cd /var/www/nexum/backend && git remote add maxfork https://github.com/maxkwaaste/nexum-backend.git && git fetch maxfork && git checkout -b feature/sdr-module maxfork/feature/sdr-module
    Or merge into the branch the server currently tracks. Max
  • Run the migration
    php artisan migrate
    Creates sdr schema + 5 tables. Isolated from all other schemas (canonical, forge, admin). Verify: php artisan tinker --execute="echo count(DB::select(\"SELECT tablename FROM pg_tables WHERE schemaname='sdr'\"));" should return 5. Claude
  • Pull frontend branch and rebuild
    cd /var/www/nexum/frontend && git fetch maxfork && git checkout -b feature/sdr-review-page maxfork/feature/sdr-review-page && npm run build && pm2 restart nexum-frontend Max
  • Verify email prerequisites exist on Hetzner DB
    php artisan tinker --execute="echo App\Models\Sync\PlatformCredential::where('platform','graph')->where('account','proxuma')->exists() ? 'Graph creds OK' : 'MISSING';"
    php artisan tinker --execute="echo App\Models\Forge\EmailSenderMailbox::where('email','max@proxuma.com')->where('is_active',true)->exists() ? 'Sender OK' : 'MISSING';"
    Both should return OK. If not, email sending will fail until fixed. Claude
  • Create SDR Sanctum token
    php artisan tinker then: $user = User::where('email', 'max@proxuma.com')->first(); echo $user->createToken('sdr-agent', ['sdr.read', 'sdr.write'])->plainTextToken;
    Store the token securely. Goes in the Claude scheduled agent config. Max
  • Add Telegram config to .env
    TELEGRAM_BOT_TOKEN=your-bot-token and TELEGRAM_CHAT_ID=your-chat-id.
    Add to config/services.php: 'telegram' => ['bot_token' => env('TELEGRAM_BOT_TOKEN'), 'chat_id' => env('TELEGRAM_CHAT_ID')] Claude
  • Clear caches and verify
    php artisan config:clear && php artisan route:clear && php artisan cache:clear
    Check schedules: php artisan schedule:list | grep sdr Claude

Option B: Full Local Copy on M1 Pro

Stand up a complete NeXum instance on M1 Pro. More setup work, but zero production risk during testing. Only choose this if you want to keep Hetzner completely untouched until after Joshua approves.

  • Install PHP 8.4 + extensions
    brew install php@8.4 + ensure pgsql, redis, bcmath, mbstring extensions. Install Composer globally. Max
  • Install and start Redis
    brew install redis && brew services start redis Max
  • Import production DB from Hetzner
    ssh hetzner "pg_dump -Fc investment_portal" > /tmp/nexum-dump.sql
    createdb nexum_local && pg_restore -d nexum_local /tmp/nexum-dump.sql
    The DB is called investment_portal (historical name). Contains canonical, forge, admin, sdr schemas. Max
  • Copy .env from Hetzner and adjust
    Critical: keep the same APP_KEY (PlatformCredential decryption depends on it). Change DB_HOST to localhost, DB_DATABASE to nexum_local, APP_URL to http://localhost:8000. Max
  • Run composer install + migrations
    cd ~/nexum/nexum-backend && composer install && php artisan migrate Claude
  • Build and serve frontend
    cd ~/nexum/nexum-frontend && npm install && npm run build && npx pm2 start ecosystem.config.cjs
    Or npm run dev for dev mode. Max
  • Serve backend
    php artisan serve --port=8000 for quick testing. For proper setup: nginx + php-fpm. Claude
  • RAM check
    After everything is running: vm_stat | head -5 and check free memory. If under 2 GB free, set Ollama keep_alive=0 to unload models after use. Max
Recommendation Go with Option A (Hetzner). The SDR migration creates an isolated sdr schema that cannot affect existing tables. Email sending is already configured. All CRM data is already there. The only risk is a broken query in SdrCandidateService (which would return a 500 error, not corrupt data). Option B adds 2-4 hours of PHP/Redis/DB setup for safety you probably don't need.

Risk Mitigation

Wrong emails go out
Every email requires manual approval in the review UI. Nothing sends automatically. The agent drafts, Max reviews. Worst case: Max approves something he shouldn't have. Mitigation: start with low-risk categories (stale deals, win-backs) before touching post-demo and inbound.
SQL query hits wrong tables
The candidate service uses raw SQL against canonical.* and forge.* schemas. If table names differ on the production DB, the endpoint returns errors. Mitigation: Phase 2 smoke test catches this immediately. Fix is a 5-minute SQL edit.
Migration conflicts with other work
The SDR migration creates its own schema (sdr.*), completely isolated from existing tables. Zero risk of conflicting with ongoing WBSO, marketing, or CRM migrations.
Joshua rejects the PR
The 3-5 day test period produces real usage data. The PR includes actual approval rates, edit patterns, and response metrics. This is stronger evidence than code review alone. If Joshua wants architectural changes, the sdr schema isolation makes refactoring contained.

Bottom Line

The code is written. Deploy to Hetzner (Option A) unless you specifically want a local sandbox. Risk is low: manual approval before any email sends, isolated sdr schema, and the existing GraphMailSendService handles Outlook integration with zero new configuration needed on Hetzner. M1 Pro (Option B) is feasible at 16 GB RAM but adds setup overhead for PHP, Redis, DB import, and APP_KEY synchronization. Start Monday May 5, test through Thursday May 8, open PR on Friday May 9.
