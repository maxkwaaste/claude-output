The NL file is the main problem: 70% of its texts are bare topic labels, not questions. The entire second half (lines 82-427) was generated as lowercase "keyword phrase?" entries. The EN file is mostly fine (91% good) with scattered long/statement issues.

Root Cause The NL reports were batch-generated with a translation script that converted EN report titles to lowercase Dutch labels rather than formulating proper Dutch questions. The EN reports were generated by Sonnet subagents that mostly wrote good questions but occasionally drifted into statements or taglines.

2. Local-to-WP Sync: Two Template Generations

The WP pages use a different HTML structure (teal hero, search-hero-title class) than the local files (blue search-hero, search-bar-text class). This means every WP page needs a full content replacement, not a patch. For ~392 reports that don't exist on WP yet, new pages need to be created under parent 3299 (EN) or 3495 (NL).

3. DAX Toggles and Copy Buttons

Verified live on proxuma.io: the DAX toggles open and show query code, the copy button works. However, prxCopyDAX() resets the button text to English "Copy Query" even in NL files where the button says "Kopieer Query". This affects all 423 NL files. The fix is a one-line change in the JS: detect the original button text and restore it instead of hardcoding English.

4. Duplicate Related-Reports Sections

17 files have the .related-reports div appearing twice (7 EN + 10 NL). This renders two identical "Related Reports" grids stacked on top of each other. Simple DOM fix: remove the second occurrence.

5. Empty Related-Report Card Descriptions

548 files (236 EN + 312 NL) have at least one .rr-card-desc that is completely empty (<div class="rr-card-desc"></div>). The card title renders fine but the description line is blank. This is likely because the batch generator didn't have the linked report's subtitle available when creating the card. Fix options:

  • Option A: Script that cross-references each linked report's .search-hero-sub text and backfills the empty descriptions. Requires matching the href slug to a local file.
  • Option B: Remove the empty .rr-card-desc div entirely so the card shows just title + category badge + link. Cleaner than showing blank space.

Execution Plan

1
Rewrite ~340 Search Bar Texts High Effort
Write a Python script that processes all 872 report HTML files and rewrites bad .search-bar-text entries. The script should:
  • NL labels (215 files): Convert bare topic labels ("autotask sales pipeline?") into natural Dutch questions ("Hoe ziet de Autotask sales pipeline eruit?"). Use the report's <h1> and .search-hero-sub as context to formulate a relevant question.
  • NL subtitles (38 files): Strip tagline phrasing ("contract winstgevendheid ranking: van held tot nul?") and rewrite as a direct question ("Welke contracten zijn het meest en minst winstgevend?").
  • EN statements (17 files): Rewrite assertions as questions.
  • EN/NL too-long (18 files): Shorten to under ~100 chars while keeping the core question.
  • Wrong language (17 files): Translate to the correct language.
  • Duplicates (3 EN files): Rephrase to be unique.
Approach: Use Claude API (batch mode) to generate replacement questions. Feed each bad text + its report title + subtitle as context. Human-review a sample of 20 before applying all. This is the biggest step -- estimated 45-60 min including review.

Quality bar: Each rewritten question should sound like something a real MSP admin would type into a search bar. Short (under 100 chars), natural language, starts with a question word. Dutch questions use proper Dutch question structure (verb-subject inversion), not translated English word order.
2
Fix Copy Button Language in NL Files Low Risk
All 423 NL files have prxCopyDAX() hardcoded to reset button text to English "Copy Query" after copying. Fix: replace btn.textContent='Copy Query' with btn.textContent='Kopieer Query' and btn.textContent='Copied!' with btn.textContent='Gekopieerd!' in all NL files. Simple sed/find-replace across the nl/ directory.
3
Remove Duplicate Related-Reports Sections Low Risk
17 files (7 EN + 10 NL) have the .related-reports div appearing twice. Write a BeautifulSoup script to detect and remove the second occurrence. Verify the first one has the correct cards before deleting.
4
Fix Empty Related-Report Descriptions Medium Risk
548 files (236 EN + 312 NL) have at least one empty .rr-card-desc. Build a cross-reference script: for each empty description, look up the linked report's slug, find the matching local HTML file, extract its .search-hero-sub text, and backfill. For cards linking to reports that don't exist yet locally, either hide the empty desc div or insert a generic one-liner based on the card title.
5
Build a Slug-to-WP-ID Mapping Low Risk
Pull the full list of existing WP pages under parents 3299 and 3495 via WP-CLI (wp post list). Map each post_name (slug) to page ID. Match these against local filenames. Categorize into: update (slug exists on WP) vs. create (new page needed). Save the mapping as a JSON file for the push script.
6
Upload Local HTML to Server Low Risk
rsync the en/ and nl/ directories to the SiteGround server (~/tmp/reports-upload/). This stages the files for WP-CLI to read. Previous sessions used the same approach successfully.
7
Push Content to WordPress Medium Risk
Write a WP-CLI batch script (run on server) that:
  • For existing pages: $wpdb->update() the post_content column (not wp_update_post() which strips styles/scripts)
  • For new pages: wp post create with --post_parent=3299 (EN) or 3495 (NL), --page_template=page-fullhtml.php, --post_status=publish
  • Derive the slug from the filename (strip proxuma-io- prefix and -post.html suffix)
  • Set proper post_title extracted from the HTML <title> tag or #prx-hero-h1
Important: Do NOT touch pages 3299, 3495, 3401, 5655, 3389, 4887 (protected). Process in batches of 50, verify each batch before proceeding.
8
Flush Cache Low Risk
Run wp cache flush && wp sg purge for dynamic cache. Then flush CDN via SiteTools socket: /chroot/tmp/site-tools.sock with domain-all API + flush_cache. This is standard procedure from previous pushes.
9
Verify Live: Visual + Functional Medium Risk
Pick 5 EN + 5 NL reports (mix of updated and newly created). For each:
  • Open in Chrome via browser MCP tools
  • Verify the search-hero question box renders correctly (blue design, question text visible)
  • Click each DAX toggle -- confirm the query code appears
  • Click "Copy Query" -- confirm clipboard gets the DAX text
  • Check morph bar, pipeline steps, KPI grid, and CTA all render properly
  • Compare side-by-side with the local file if anything looks off
If SG optimizer is stripping JS or styles, the fix is to add the pages to SG's exclusion list or adjust the minification settings.
10
Update Gallery Pages (3299 + 3495) Medium Risk
The gallery pages (EN 3299, NL 3495) use ACF field bf_sections_0_html_code for their content. Regenerate gallery HTML to include all ~449 EN and ~423 NL report cards with correct links. Update via update_post_meta(). Gallery cards should link to the correct WP slugs.

Risk Assessment

SiteGround Optimizer SG optimizer has been known to silently blank pages larger than ~5MB (inline base64 images) and can interfere with inline JS through minification. The report files are 30-150KB each, well under the size limit. But JS minification could break the DAX toggle event listeners. If this happens, the fix is to exclude /powerbi/insights/* from SG's JS combine/minify settings.
The $wpdb->update() Requirement For existing pages, we must use $wpdb->update() directly, not wp_update_post(). The latter runs content through wp_kses and strips <style>/<script> tags. This is a documented gotcha from previous sessions. For new pages, wp post create via WP-CLI is safe because WP-CLI bypasses kses for admin users.
