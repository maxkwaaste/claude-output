Pillar 1: Transcript Pipeline

Problem Today someone has to press "Record" in Teams, then manually export the .vtt transcript, then drop it into a Claude session. Consultants won't do this consistently. The system breaks at step one.

Prerequisite: Teams Admin Policy

The Dxfferent M365 tenant needs a one-time policy change in Teams Admin Center: enable automatic recording and automatic transcription for all users (or a specific user group). Auto-recording causes a notification banner for all meeting participants. This is a compliance consideration -- external attendees will see "Recording has started."

Three Approaches

Approach How It Works Pros Cons
A: Graph API Polling Daily scheduled job on M1 Pro calls Microsoft Graph API to fetch new transcripts from the last 24 hours No extra licensing. Full control. Runs on existing M1 Pro automation infra. Up to 24-hour delay. Requires specific Graph API permissions (OnlineMeetings.Read.All).
B: Power Automate Power Automate triggers on "When a meeting recording is available," copies transcript to OneDrive folder Event-driven, near real-time. Microsoft handles trigger reliability. Another moving part. Harder to customize. Power Automate debugging.
C: Hybrid (Recommended) Power Automate handles trigger and file organization. M1 Pro watches OneDrive folder for AI processing. Each system does what it's best at. Clean boundary (a folder of files). Easy to debug. Two systems to maintain.

Pillar 2: AI Extraction & CRM Enrichment

Problem Transcripts are raw text. Forge CRM needs structured data: pain points, deal stages, contact profiles, action items. Today Max does this manually, one transcript at a time.

Call Classification

Type Identified By Processing
Sales demo/pitch Max is on the call + sales language (pricing, demo, features) Full sales extraction
Consulting intake Joshua + new customer + scoping language Consulting intake extraction
Consulting session Any consultant + existing customer + project language Consulting session extraction
Internal All attendees are @dxfferent.com or @proxuma.io Skipped

Sales Call Extraction

Pain Points
Verbatim quotes from the prospect, one-sentence summary, severity rating (high/medium/low). Accumulated over time, never overwritten.
Objections
Categorized (budget, timing, internal approval, competitor, skepticism, technical). Includes how Max handled it and whether it was resolved.
Deal Stage
AI recommends: discovery, qualified, proposal, negotiation, won, lost, stalled. Auto-advance allowed up to "qualified" only. Beyond that requires Max's approval.
Buying Signals
Statements indicating purchase intent, extracted verbatim from the conversation.
Call Quality Score
1-10 against the sales framework defined in the playbook (Pillar 3). Coaching notes: what went well, what to improve.
Next Steps
Action items with owner (Max or prospect). Deadlines if mentioned in conversation.

Consulting Call Extraction

Project Progress
Topics discussed, matched to existing project in Forge or flagged as new.
Environment Updates
Tech stack changes, new tools, new info about the customer's setup.
Action Items
List with owner and deadline. Both sides (consultant + customer).
Satisfaction Signals
Positive, negative, or neutral with evidence quotes. Blockers preventing progress.

CRM Write Logic

  • Contact matching: Email address from metadata.json as primary key. No match = draft contact flagged for review.
  • Company matching: Attendee email domain + existing Forge records. No match = draft company flagged for review.
  • Data writing: Structured "call intelligence" note on the contact/company record, timestamped, linked to transcript and recording on OneDrive.
  • Pain points and objections: Accumulated over time on the company record. Appended, never overwritten.
  • Deal stage: Auto-advance up to "qualified." Beyond that = proposed update, Max approves or rejects. Reasoning always included.

Processing Schedule

Daily batch at 07:00 CET on M1 Pro (100.85.64.20). Morning summary to Max: calls processed (by type), internal calls skipped, deal stage changes, new draft contacts/companies, any failures.

Pillar 3: Sales Playbook Generation

Problem Max's sales knowledge lives in his head. When he hires a closer or SDR, there's no documented process for how to demo Proxuma/Dxfferent, handle objections, or qualify leads.

Phase 1: Initial Generation

AI analyzes the full library of 140+ demo transcripts in Open Brain to produce:

Sales Process Map
How Max opens calls, transitions to demo, order of feature demos, pricing conversations, close or next-step setup. Talk ratio and average call length.
Objection Library
Every objection categorized with frequency percentages. Verbatim examples of successful and unsuccessful responses. Which responses led to positive outcomes vs. stalls. Proxuma's version of Leonard's "74% budget" discovery.
Qualification Criteria
What closed-won customers have in common. What stalled/lost deals share. Red flags that show up early. Green flags that predict conversion.
Product Positioning
Features that get the strongest reactions. Features that confuse. How Max actually explains the value prop (real words, not marketing copy). Most frequent prospect questions.

Phase 2: Human Review

Max reviews each section with four responses:

  • "This is right" -- keep as-is
  • "I do this but I shouldn't" -- flag for improvement, note what to do instead
  • "The AI misread this" -- correct with actual intent or context
  • "This is missing" -- add knowledge the transcripts don't capture

Phase 3: Living Document

  • New objection types added automatically as they appear
  • Call quality scores measured against the playbook's framework (the playbook IS the scoring criteria)
  • Quarterly review: AI generates a changelog of what shifted
  • Top-scored calls flagged as training examples for future hires
  • New hire onboarding: playbook + top 10 calls to watch + their calls scored against the same framework

Playbook lives in Forge (Flow Nexus) as a structured document. Sections are independently updatable. Every claim links back to transcript evidence. Version history maintained.

Pillar 4: Sales Claw

Problem Deals fall through the cracks. Max is the only salesperson managing ~200 open deals. Prospects promised a proposal don't get one. Customers with dropping satisfaction don't get a check-in. NemoClaw proved the concept on Twenty CRM but hit limitations: Anthropic removed shell access, the draft queue overflowed, and it was tightly coupled to the wrong CRM.
Replaces NemoClaw Sales Claw is a clean rebuild for Forge CRM. Model-agnostic (Claude or GPT). Runs 3x/day via cron jobs, not an always-on daemon. Carries forward NemoClaw's proven patterns: draft diff engine, pre-call briefings, and edit-based learning.

Architecture: Three Loops

1
Scoring Loop
Score every open deal on urgency, produce ranked list
2
Action Loop
Draft emails for top N deals (adaptive cap)
3
Learning Loop
Edit diffs + feedback + patterns improve over time

Scoring Signals

Staleness
Days since last activity vs. stage norms. A proposal-stage deal going quiet for 5 days is more urgent than a new-stage deal going quiet for 5 days.
Broken Promises
Next steps from Pillar 2 call intelligence: did they happen? Prospect asked for pricing -- was it sent? Max said "I'll send X by Friday" -- did he?
Inbound Urgency
New leads with no response (highest priority). Prospects who replied but haven't heard back from Max.
Momentum
Deal stage progression speed vs. average. Recent engagement frequency vs. historical. Buying signals from last call.
Satisfaction (Consulting)
Satisfaction drops between consecutive calls. Unresolved blockers accumulating. From Pillar 2 consulting extraction data.
Composite Score
0-100 per deal. Suggested action: email, call, send_proposal, send_offer, or no_action. Weights shift via Learning Loop.

Draft Review Interface

Purpose-built in Forge CRM. Every action feeds the Learning Loop. Max never opens Outlook for Sales Claw drafts.

Action What Happens What Claw Learns
Approve & Send (A) Email sends via MS365 MCP immediately Draft was good -- reinforces style, timing, prospect selection
Edit & Send (E) Max edits inline, hits send Before/after diff captured. Patterns extracted from edits
Reject (R) Draft discarded Wrong prospect or approach -- weakens similar scoring
Feedback (F) Free-text box, dismiss or re-draft Explicit correction as labeled training data
Snooze (S) Moves to tomorrow's queue Right prospect, wrong timing -- adjusts staleness threshold

Keyboard shortcuts in parentheses. Target: 5 drafts reviewed in under 2 minutes. Adaptive cap starts at 5/day. If Max sends 4/5, cap rises. If Max ignores 3/5, cap drops.

Learning Loop

Channel 1: Edit Diffs (passive)
Highest signal. Every Edit & Send produces a before/after pair. After 20+ pairs, Claw generates "Max's writing rules" -- specific, stored in Forge, human-editable.
Channel 2: Telegram Feedback (explicit)
Free-text corrections from review interface. Telegram reserved for escalation: unreviewed drafts, urgent inbound, satisfaction drops.
Channel 3: Action Patterns (implicit)
Approve rate, time-of-day patterns, prospect-type preferences, volume tolerance. Auto-adjusts cap and schedule.
Visibility
"Claw is learning" page in Forge: drafts reviewed, approve/edit/reject ratios over time, top writing rules, scoring weight changes. Builds trust.

Nudge Dashboard

Single view in Forge CRM -- Max's daily sales cockpit. Three sections:

  1. Today's Queue: Draft review cards sorted by urgency, keyboard-navigable, progress indicator ("3 of 5 reviewed")
  2. Pipeline Health: All open deals color-coded (green/yellow/red), filterable by stage/health/action type, expandable to full deal context + engagement timeline
  3. Alerts: Unresponded inbound leads, broken promises, satisfaction drops, returning prospects

Execution Architecture

Scheduler (cron / launchd / Flow Nexus / Claude scheduled agents) | v Runner Script (Node.js or Python) - Reads config (schedule, cap, weights) - Calls Forge CRM API for pipeline data - Calls LLM API with tools for scoring + drafts - Writes results back to Forge CRM - Sends via MS365 MCP when Max approves | v LLM Adapter (swappable) - Claude API (Anthropic SDK) - OpenAI API (GPT / OpenClaw compatible) - Model choice in config, not code | v Learning Store (Forge CRM) - Edit diffs, feedback records - Writing rules (generated + manual) - Scoring weights (auto-adjusted)

Schedule: 3 runs/day -- 07:00, 12:00, 16:30 CET. Configurable. Communication: Forge review interface (primary), Telegram (escalation only), Outlook (sending mechanism via MS365 MCP).

Key Insight

AI sees patterns across every call that no human has the bandwidth to track

Leonard discovered that 74% of his objections were budget-related -- something he missed during months of manual reviews. This system should surface the equivalent for Proxuma. With 140+ transcripts already in Open Brain, the data is there. It just needs structured extraction.

Open Questions

  1. Morning summary channel: Email, Teams message, or Claude session?
  2. Forge CRM schema: Exact data model for contacts, companies, deals, and call intelligence notes needs to be defined during implementation.
  3. Outlook-to-Forge connection: When live, contact/company matching may simplify. Spec works without it.
  4. Historical backfill: Reprocess 140+ existing transcripts through this system to populate Forge retroactively? Or start fresh?
  5. Consultant buy-in: The system records all their calls. They need to know and agree.
  6. Call quality scoring threshold: At what score should a call be flagged? Needs calibration after playbook generation.
  7. Graph API transcript access: Verify current MS365 MCP permissions cover meeting transcript retrieval.
  8. Scoring weight calibration: Initial weights need tuning against historical data. Adjust based on first 2 weeks of scored output.
  9. Learning loop cold start: ~20+ edit-diff pairs needed before writing rules become useful. Static style template until then.
  10. NemoClaw data migration: Migrate historical learning data to Forge, or start fresh?
  11. MS365 send vs. draft: Can MS365 MCP send directly, or only create drafts? Affects review interface design.
